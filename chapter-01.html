Chapter 01 Introduction

What is Claude Code?

Claude Code is a command-line tool that embeds Claude directly into your development workflow. Unlike a chat interface, it operates autonomously within your terminal — reading files, running commands, editing code, managing git, and coordinating multiple specialized agents — all to complete engineering tasks end-to-end.

bolt The Agentic Shift

Traditional AI coding tools operate in a request-response loop. You write a prompt, get a suggestion, accept or reject it. Claude Code operates differently.

warning Traditional AI Coding Assistant

You ask a question. It responds. You apply the change manually. Every step requires your explicit action. The model has no knowledge of your filesystem, running processes, or git state unless you paste it in.

check_circle Claude Code (Agentic)

You describe a goal. Claude Code reads your codebase, plans a sequence of actions, edits files, runs tests, checks git status, and reports back with results — all autonomously. You review the outcome, not every step.

This distinction has direct implications for how you work: rather than crafting precise prompts for individual edits, you define goals and constraints, then let Claude Code figure out the path.

key Core Mental Model

Claude Code is not a smarter autocomplete. It is an autonomous agent that takes multi-step actions. The shift from "assistant that suggests" to "agent that executes" is the single most important reframe when working with it.

Under the Hood

How Claude Code Works

Claude Code is not just a language model with file access. It operates through a structured agentic loop — a repeating cycle of perception, planning, action, and observation that continues until the task is complete.

cycle The Perceive → Plan → Act → Observe Loop

Every Claude Code interaction — whether a single command or a multi-hour task — runs through the same four-phase cycle:

1
Perceive

Claude Code reads its inputs: your prompt, the CLAUDE.md memory files, conversation history, and any files or tool outputs already in context. This is what the model "knows" before it acts.

2
Plan

The model reasons about what tools to invoke, in what order, and with what parameters. It decides: "I need to read auth.go first, then check the test file, then make the edit." This planning happens as internal reasoning before any tool is called.

3
Act

Claude Code executes one or more tool calls: reading a file, writing a diff, running a shell command, or spawning a subagent. Each tool call produces a result that flows back into the context.

4
Observe

The result of the action is appended to the context window. Claude Code reads the output — a file's contents, a test failure, a build log — and decides whether to loop again or report completion.

Note: The loop continues autonomously until Claude Code determines the task is done or it hits a decision point that requires your input (such as a permission prompt or ambiguous requirement). You are reviewing outcomes, not steering every step.

build Tool Use Mechanics

Claude Code interacts with your system exclusively through a defined set of tools. It cannot do anything outside these tools — which is also what makes it auditable and safe to run with permission boundaries.

Tool CategoryWhat It DoesExamples
File ReadReads file contents into contextReading source files, configs, specs
File WriteCreates or overwrites filesGenerating new files, writing tests
File EditTargeted string replacements in existing filesBug fixes, refactors, adding imports
BashRuns shell commandsBuild, test, lint, git operations
Glob / SearchFinds files by pattern or contentLocating all usages of a function
SubagentSpawns a focused child agentCode review, security audit
MCP ToolsExternal integrationsBrowser, GitHub, databases
Tip: When Claude Code asks for permission before running a command, it is because that tool call has a side effect — it writes to disk, runs a process, or makes a network request. Read-only operations (file reads, searches) typically do not require explicit approval.
Beginner Walkthrough

First Steps with Claude Code

The first time you run Claude Code in a real project, the pattern that works best is: orient, then ask questions, then give tasks. Do not start by asking it to rewrite something — start by asking it to explain.

terminal Your First Commands

Open your terminal, navigate to a code project you know well, and run:

bash 
# Step 1 — enter the REPL
claude

# Step 2 — let Claude Code orient itself (ask a question first)
What does this project do and how is it structured?

# Step 3 — go deeper on something specific
Walk me through how a request flows through the authentication middleware.

# Step 4 — give a small, concrete task
Add input validation to the CreateUser handler. The email field must be a valid email address.
Tip: Asking Claude Code to explain your codebase before giving it tasks accomplishes two things: it confirms context loading is working correctly, and it surfaces any misunderstandings before they turn into incorrect edits. Treat it like onboarding a new developer — orient first.

question_mark Asking Questions vs. Giving Tasks

Claude Code handles both questions and tasks, but they have different properties. Understanding the distinction helps you use each effectively.

✓ Questions (Safe to Ask Freely)

prompt
What are all the places this function is called?
Why does this test fail?
What would break if I changed this interface?
How does the retry logic work in this service?

Questions only read — they never write. Use them freely to build understanding without risk.

✗ Vague Tasks (Avoid)

prompt
Improve the code.
Fix the tests.
Refactor this.
Make it better.

Vague tasks lead to unpredictable changes. Claude Code will do something — but not necessarily what you meant.

Warning: Unlike asking a question to a human colleague, asking Claude Code to "refactor this file" will immediately start making changes. There is no soft intent to infer — Claude Code takes action. Be specific about what you want changed and why.

loop The Iterative Workflow

Claude Code works best iteratively. Break large goals into stages, verify each stage before moving forward.

1
Start with Plan Mode

Press Shift+Tab to enter read-only Plan Mode. Describe what you want. Review the proposed approach before any files are touched.

2
Execute in Small Steps

Give Claude Code one well-defined task at a time. After each step, run the tests or build to verify nothing broke before continuing.

3
Review Changes with Git

After each set of edits, run git diff to review what changed. Claude Code's changes are real code — review them as you would any PR.

4
Commit and Continue

Commit working increments frequently. This gives you safe recovery points and makes it easy to revert a Claude Code change that did not work as expected.

Best Practices

Best Practices for Getting Started

These patterns consistently produce better results with Claude Code. They apply to beginners and experienced users equally — they encode the lessons that take most people a week of usage to discover.

edit_note Writing Good Prompts

A good Claude Code prompt has four elements: context, goal, constraints, and output format. You do not need all four every time, but including them reduces back-and-forth significantly.

✓ Specific and Bounded

prompt
In internal/handlers/user.go, add input
validation to the CreateUser function.
Validate that email is a valid email
format and name is non-empty.
Return a 400 with a JSON error body.
Do not change the handler signature.

✗ Vague and Unbounded

prompt
Add validation to the user handler.

The Constraints Habit

Always include what Claude Code should not do. Constraints are as important as goals — they prevent well-intentioned scope creep that turns a 5-file task into a 20-file refactor.

Tip: End prompts with explicit scope limits: "Only change files in internal/handlers/." or "Do not modify the database schema." or "Do not add new dependencies." These are not obvious to the model — make them explicit.

account_tree Breaking Down Large Tasks

Large tasks produce worse results than small tasks. When a task touches many files or requires multiple decisions, break it into sequential phases:

prompt — phased approach 
# Phase 1 — understand before acting
Read the existing auth implementation and describe how tokens are validated.

# Phase 2 — plan
[Shift+Tab to Plan Mode]
I want to add refresh token support. What changes are needed and in which files?

# Phase 3 — implement incrementally
Implement step 1 from your plan: add the RefreshToken field to the User model.

# Phase 4 — verify before continuing
Run the tests for the auth package and tell me if anything fails.

cleaning When to Use /clear

Context window health is a real concern in long sessions. These are the signals that tell you it is time to run /clear:

SignalWhat It MeansAction
Claude Code "forgets" a constraint you set 30 turns agoThat instruction has been pushed out of attention by volume/clear and restate it
Responses start referencing earlier (now irrelevant) codeContext confusion — old context polluting new task/clear
You are starting a completely different taskDifferent goal = different context/clear or new session
A hallucination crept into the conversationContext poisoning — bad info now in history/clear immediately
Pitfalls

Common Mistakes

These are the patterns that cause the most friction for new Claude Code users. Each is avoidable once you know to watch for it.

report_problem The Most Common Anti-Patterns

Warning — Too Vague: "Improve the codebase" is not a task. Claude Code will make changes, but they will reflect its judgment, not your intent. Always specify the file, the behavior, the constraint, and the reason.
Warning — Too Much at Once: Asking Claude Code to implement an entire feature in one prompt degrades quality. The model loses precision as scope expands. Break it into phases: model first, then API, then tests, then documentation.
Warning — Not Reviewing Changes: Claude Code writes real code. Review it with git diff before committing, the same way you would review any contributor's PR. Autonomous does not mean infallible.
Warning — Reusing Sessions Across Tasks: Running a debugging session and then immediately asking for a new feature in the same session allows the debugging context to bleed into the feature work. Use /clear between unrelated tasks.

✓ Effective Task Framing

prompt
In pkg/auth/token.go, the ValidateToken
function returns true for expired tokens
when the clock skew is > 30s.
Fix the comparison on line 47 so
expiry is checked against the adjusted
time. Do not change the function
signature or add new dependencies.

✗ Common Anti-Pattern

prompt
The token validation is broken, can you
fix it and also improve the auth code
generally while you're in there?
Use Cases

When to Use Claude Code

Claude Code is most effective for tasks with clear inputs and verifiable outputs. Understanding where it excels — and where it is less suited — helps you integrate it effectively into your workflow.

check_circle High-Leverage Scenarios

search

Codebase Exploration

Onboarding to a new codebase. "Explain how request routing works in this service" saves hours of manual tracing through unfamiliar code.

bug_report

Debugging

Paste a stack trace and ask Claude Code to trace the failure path through the actual source files. It reads the relevant code rather than guessing.

science

Test Generation

Generate unit and integration tests for existing code. Provide the implementation and the testing framework — Claude Code writes coverage that matches your patterns.

swap_horiz

Repetitive Refactors

Rename a function across 40 files. Update all callers to a new API signature. Convert a pattern consistently across a large codebase. These take seconds with Claude Code.

rate_review

Code Review Assistance

Before submitting a PR, ask Claude Code to review the diff for edge cases, missing error handling, or security issues. A second set of eyes that never gets tired.

description

Documentation

Generate README sections, API documentation, and inline comments from the actual implementation — not from memory or guesswork.

Note: Claude Code is less suited for tasks with no verifiable output — subjective architectural decisions, product strategy, or anything that requires deep domain knowledge that is not captured in the codebase. Use it where the correctness signal is clear: tests pass, build succeeds, the diff looks right.
What It Can Do

Core Capabilities

Claude Code integrates deeply with your development environment across the entire software development lifecycle — from initial design through deployment and monitoring.

folder_open

Codebase Understanding

Reads and reasons across your entire project. Understands architecture, dependencies, naming conventions, and existing patterns before making changes.

build

File & Terminal Operations

Creates, edits, deletes files. Runs shell commands, builds, tests, and linters. Interacts with git, docker, npm, and any CLI tool in your environment.

psychology

Context Engineering

Manages its own memory through CLAUDE.md files, /compact and /clear commands, and a three-tier memory hierarchy that persists across sessions.

smart_toy

Multi-Agent Orchestration

Spawns specialized subagents — code reviewers, test writers, security auditors — each with its own focused context. The main instance acts as orchestrator.

power

MCP Integrations

Connects to external tools via the Model Context Protocol. Browser automation (Playwright), GitHub, databases, Slack, and any custom MCP server.

settings

Hooks & Automation

Executes custom shell commands on lifecycle events (pre-tool, post-tool, session stop). Enables automation pipelines, audit logging, and workflow integration.

shuffle Multi-Agent Workflows

One of the most powerful capabilities is the ability to orchestrate fleets of specialized agents. Rather than doing everything in a single context window — which degrades quality as complexity grows — Claude Code delegates:

1
Task Decomposition

The main agent breaks a complex task into independent subtasks that can be parallelized or sequenced.

2
Subagent Dispatch

Specialized subagents (code reviewer, security auditor, test writer) are spawned with focused contexts — each has exactly the information it needs, no more.

3
Result Synthesis

The orchestrating agent collects outputs, resolves conflicts, and assembles the final result without any individual agent's context becoming polluted.

lightbulb Key Insight

Context isolation is not just an optimization — it is a quality guarantee. A code reviewer that only sees the diff it needs to review will produce sharper feedback than an agent that has been doing refactoring, debugging, and documentation all session.

Before You Start

Prerequisites

Claude Code is a developer tool for working software engineers. It assumes you are comfortable with your terminal, understand version control, and can read code in at least one language. No machine learning background required.

assignment Required Knowledge

AreaLevelNotes
Programming languageRequiredPython or TypeScript/JavaScript fluency. Can read and write production code, not just scripts.
GitRequiredBranching, committing, pushing. Claude Code interacts with git constantly.
Terminal / CLIRequiredComfortable running commands, reading output, setting environment variables.
Virtual environmentsRequiredvenv, nvm, or equivalent. Required for clean dependency management.
Node.js / npmRecommendedClaude Code is installed via npm. Node.js 18+ required.
DockerRecommendedUseful for sandboxed execution and MCP server containers.
Machine learningNot RequiredNo ML background needed. You are using the model, not training it.
warning Audience Note

Claude Code is designed for software engineers, data scientists, and experienced developers who write production code regularly. If you are new to programming, focus on language fundamentals first — Claude Code is significantly more useful once you can evaluate the quality of its output.

desktop_mac System Requirements

  • Operating system: macOS, Linux, or Windows with WSL2
  • Node.js: v18 or later — check with node --version
  • npm: v8 or later (bundled with Node.js)
  • Anthropic API key: Required — obtain from console.anthropic.com
  • Network access: Claude Code makes API calls to Anthropic's servers per request
Setup

Installation

Claude Code is distributed as an npm package. Install it globally so it is available from any project directory in your terminal.

rocket_launch Install & Authenticate

bash 
# Install Claude Code globally
npm install -g @anthropic-ai/claude-code

# Verify installation
claude --version

# Authenticate with your API key
claude auth login

After running claude auth login, you will be prompted to enter your Anthropic API key. It is stored securely in your system keychain — you will not need to re-enter it per session.

Initialize in a Project

Navigate to any code repository and run /init to have Claude Code analyze your project and generate a CLAUDE.md context file:

bash 
# Navigate to your project
cd /path/to/your/project

# Launch Claude Code REPL
claude

# Inside the REPL — initialize project context
/init

The /init command traverses your directory structure, identifies the tech stack, infers architectural patterns, and writes a CLAUDE.md file that is automatically loaded in all future sessions.

lightbulb Best Practice

Commit your CLAUDE.md to version control. It serves as shared context for every team member using Claude Code on the project — equivalent to an always-loaded engineering brief that every AI session reads first.

Verify the Setup

bash 
# Quick sanity check — one-shot mode (no REPL)
claude -p "Describe the architecture of this project in 3 sentences"

# Expected: coherent description of your stack and structure
# Not expected: confusion or hallucinated file paths

If the response accurately reflects your project's structure, context loading is working correctly and you are ready to start.
