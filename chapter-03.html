Chapter 03 Getting Oriented

The Claude Code Interface

Claude Code has three primary interaction modes. Each serves a different workflow stage — knowing which to reach for and when is fundamental to working efficiently.

Three Interaction Modes

Interactive REPL

The default mode. Run claude in your terminal to enter the multi-turn REPL. Full tool access, slash commands, memory loading, and interactive permission prompts.

One-Shot / Headless

Run claude -p "your task" to execute a single prompt and exit. Ideal for scripts, CI pipelines, and automation where you need Claude Code without a human in the loop.

Plan Mode

Activate inside the REPL with Shift+Tab. Read-only — Claude Code can read files and reason, but makes no changes. Use for research, architecture review, and planning before execution.

bash
# Start interactive REPL
claude

# One-shot: run a task and exit (no REPL)
claude -p "Add error handling to all HTTP handlers in internal/handlers/"

# Continue the most recent conversation
claude --continue

# Resume a specific session by ID
claude --resume abc123
First Steps

Project Initialization with /init

The /init command is the entry point for any new project. It performs a one-time analysis of your codebase and generates a CLAUDE.md that becomes the foundation of your project's memory.

What /init Does

1
Traverses directory structure

Reads file names, folder structure, and key configuration files (package.json, go.mod, Makefile, etc.) to understand the project shape.

2
Identifies the tech stack

Detects frameworks, languages, build systems, test runners, and CI configuration based on the files present.

3
Infers architecture

Reads representative source files to understand naming conventions, design patterns, and structural organization.

4
Writes CLAUDE.md

Produces a structured context file with architecture summary, key directories, build/test commands, and inferred conventions.

After /init

Read and edit the generated CLAUDE.md before committing it. Claude Code's inferences are a good starting point but may miss important conventions, constraints, or off-limits areas that are implicit in your team's culture. Make them explicit here.

Core Workflow

Plan Mode

Plan mode is a read-only state where Claude Code can explore your codebase, reason about architecture, and produce a plan — without modifying any files. It is the right tool for the design phase of any non-trivial task.

⌨ Activating Plan Mode

Inside the REPL, press Shift+Tab to toggle between normal mode and plan mode. A visual indicator in the interface confirms which mode is active.

What Plan Mode Can Do

  • Read any file in your project
  • Traverse the directory tree
  • Run read-only shell commands (e.g., git log, go list)
  • Reason about architecture and produce a structured plan
  • Generate todo lists for multi-step tasks

What Plan Mode Cannot Do

  • Write or modify files
  • Run commands with side effects
  • Make network requests (unless explicitly permitted)
When to Use Plan Mode

Use plan mode at the start of any task that touches more than 3-4 files, involves architectural decisions, or has significant risk of unintended side effects. A 10-minute planning phase reliably produces better outcomes than jumping straight to execution — especially for refactors and cross-cutting changes.

Development Pattern

Spec-Driven Development

Before asking Claude Code to implement anything non-trivial, write a spec file. A spec is a structured document that defines what you are building, why, and how — before a single line of implementation code is written.

Why Specs First

Specs serve two purposes simultaneously:

  • For you: Forces clarity on requirements before implementation. Catches ambiguities and scope issues before they cost implementation time.
  • For Claude Code: Provides a stable, structured reference that can be loaded with @ syntax in any session. More reliable than inline instructions that age with the conversation.

Spec File Structure

markdown — spec/feature-name.md
# Feature: Hook Registry MVP

## Overview
A web interface for browsing and sharing Claude Code hooks.
Users can search by category, view hook details, and copy the hook configuration.

## Scope
MVP only: read-only browsing. No auth, no user accounts, no submissions.

## Data Model
Hook: { id, name, category, description, config, author, tags }
Categories: pre-tool | post-tool | notification | stop

## UI/UX Requirements
- Grid layout on desktop, list on mobile
- Filter by category
- Search by name and description
- Copy-to-clipboard for hook config JSON

## Technical Architecture
Next.js App Router, Tailwind CSS, static JSON data source
No database for MVP — data in /public/hooks.json

## Out of Scope
- User submissions
- Authentication
- Comments or ratings

Using the Spec in Sessions

prompt
Implement the Hook Registry per @spec/hook-registry.md.
Start with the data model and the search/filter logic.
Spec + Fresh Session

The most reliable implementation pattern: write the spec, commit it, then start a fresh Claude Code session and load the spec with @. The fresh session has no accumulated context residue, and the spec provides a complete, self-contained brief for the entire task.

Configuration

Model Selection

Claude Code supports multiple model tiers. Choosing the right model for a given task affects both quality and cost — some tasks justify the most capable model, others do not.

Available Models

ModelBest ForTrade-off
Claude Opus 4 Deep reasoning, architecture planning, complex refactors, spec writing Higher cost and latency — reserve for tasks that genuinely need it
Claude Sonnet 4 Day-to-day implementation, code review, test writing, most tasks Best balance of quality, speed, and cost for typical engineering work
Claude Haiku 4 Simple edits, formatting, repetitive tasks, batch operations Fastest and cheapest — appropriate for high-volume low-complexity work

Switching Models

prompt — inside REPL
# List available models
/models

# Set model for this session
/model claude-opus-4-5

# Start with a specific model from CLI
claude --model claude-sonnet-4-5
Multimodal Support

All Claude Code models accept images and screenshots as input. You can paste a screenshot of a UI design, a diagram, or an error message directly into the REPL or reference it via @ path. This is useful for implementing designs from mockups or debugging visual issues.

Context Hygiene

Session Management

Every turn in a Claude Code session accumulates context. Left unmanaged, this degrades output quality over time. Effective session management is as important as good prompts.

The Session Lifecycle

A Claude Code session goes through predictable phases:

  • Fresh start: Memory files loaded, context window nearly empty, highest quality
  • Working phase: File reads, edits, and tool outputs accumulate — quality stable
  • Degradation zone: History grows large, responses become less precise, instructions from early in the session may be "forgotten"
  • Intervention point: /clear, /compact, or new session

Practical Session Patterns

One Task, One Session

The simplest and most reliable session strategy: start a new session for each major task. Load your spec or CLAUDE.md context files, do the task, exit. Don't reuse sessions across unrelated work — the accumulated context from one task actively hurts the next.

workflow — managing long sessions
# Midway through a long task — history getting noisy
/compact                      # Compress to essentials, keep going

# Switching to a different task in the same codebase
/clear                        # Wipe history, memory files reload automatically

# Done for the day, want to resume tomorrow
# Session ID is shown in /status — note it
/status
/exit

# Next day — resume with full history restored
claude --resume <session-id>
Integrations

Connecting MCP Tools

Claude Code connects to external tools through the Model Context Protocol. MCP servers extend what Claude Code can do — browser automation, GitHub access, database queries, and more.

Installing MCP Servers

bash
# Install Playwright MCP — browser automation
claude mcp install @playwright/mcp@latest -s user

# Install at project scope (only this project)
claude mcp install @playwright/mcp@latest -s project

# List installed MCP servers
claude mcp list

# Remove an MCP server
claude mcp remove playwright

Scope: User vs Project

-s user (User Scope)

Available in all Claude Code instances for this user. Installed once, works everywhere. Good for general-purpose tools (Playwright, filesystem, GitHub).

-s project (Project Scope)

Available only in this project directory. Stored in .claude/settings.json and committed to version control — all team members get it.

Playwright Example

After installing Playwright MCP, Claude Code can interact with browsers — taking screenshots, clicking elements, filling forms, and scraping content:

prompt
Open localhost:3000 in a browser, take a screenshot of the homepage,
and identify any layout issues at mobile viewport (375px wide).
Context-Specific Memory Files

Create tool-specific context files like memory/playwright/CLAUDE.md with guidance on how to use the tool effectively in your project (base URL, auth patterns, selector conventions). Load them explicitly with @ when doing browser-related tasks rather than including them in the root CLAUDE.md.

Command Reference

Slash Command Reference

Slash commands are typed directly in the REPL prompt. They control session state, model configuration, context, and tool management. Knowing these commands well significantly improves your day-to-day workflow.

Essential Commands

/clear/clearWipes conversation history. CLAUDE.md files are preserved and reloaded.Starting a new task, context is polluted

/status/statusShows current session ID, model, token usage, and loaded memory filesBefore exiting if you want to resume later

Keyboard Shortcuts

The Claude Code REPL has a set of keyboard shortcuts for common actions. Learning these avoids repetitive typing for the operations you perform every session.

REPL Shortcuts

Ctrl+CInterrupt / CancelStops a running tool call or in-progress generation. Safe to use mid-execution.
↑ / ↓History navigationCycles through previous prompts in the current session

Configuration

Configuration Deep Dive

Claude Code's behavior is controlled through .claude/settings.json (project scope) and ~/.claude/settings.json (user scope). Understanding these settings lets you configure permissions, hooks, and MCP tools for your environment.

settings.json Structure

json — .claude/settings.json (annotated) 
{
  "model": "claude-sonnet-4-5",

  "permissions": {
    "allow": [
      "Bash(go test ./...)",
      "Bash(make *)",
      "Bash(git diff*)",
      "Bash(git log*)"
    ],
    "deny": [
      "Bash(git push*)",
      "Bash(rm -rf*)"
    ]
  },

  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "echo 'Running: $CLAUDE_TOOL_INPUT' >> ~/.claude/audit.log"
          }
        ]
      }
    ],
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "notify-send 'Claude Code finished'"
          }
        ]
      }
    ]
  },

  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": ["@playwright/mcp@latest"],
      "scope": "project"
    }
  }
}

Permission Rules

The permissions block uses glob-style patterns to pre-approve or block tool calls. Pre-approved commands run without an interactive prompt:

PatternMatchesUse Case
Bash(go test ./...)Exact command onlyPre-approve your test command
Bash(make *)Any make targetAllow all Makefile targets
Bash(git diff*)Any git diff variantAllow read-only git operations
Bash(git push*) in denyAny git pushBlock pushes — require manual review
Warning: Permissions in the project .claude/settings.json apply to every developer who uses Claude Code in that project. Set project-level permissions to allow safe operations (build, test, lint) and block irreversible ones (push, database drops). Individual developers can extend their personal permissions in ~/.claude/settings.json.

Hook Configuration

Hooks execute shell commands on lifecycle events. They are the mechanism for workflow automation, audit logging, and CI/CD integration.

Hook EventWhen It FiresCommon Use
PreToolUseBefore any tool callAudit logging, command validation
PostToolUseAfter any tool call completesRun linter after file edits, notify on completion
StopWhen the session endsPlay a sound, send a notification, run cleanup
NotificationWhen Claude has a question or needs inputDesktop notification when Claude is waiting for you
json — hook: run lint after every file edit 
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "golangci-lint run --fix 2>&1 | head -20"
          }
        ]
      }
    ]
  }
}
Workflows

Workflow Patterns

These are concrete multi-step workflows that combine Claude Code's features. Each addresses a common engineering scenario and shows the command sequence to execute it.

Pattern: New Feature from Spec

The most reliable pattern for implementing a non-trivial feature from scratch.

workflow — new feature 
# 1. Write spec before opening Claude Code
#    Create spec/feature-name.md with requirements, data model, constraints

# 2. Start fresh session
claude

# 3. Enter Plan Mode before touching any code
#    Press Shift+Tab, then describe the task
Read @spec/feature-name.md and produce an implementation plan.
List affected files and describe each change needed.

# 4. Review the plan, then exit Plan Mode (Shift+Tab again)
# 5. Implement step by step, verifying each step
Implement step 1: create the data model in internal/models/feature.go

# 6. Run tests after each step
Run the tests for the models package and fix any failures.

# 7. Commit working increments
Stage all changes and create a commit with message: "feat: add feature-name model"

Pattern: Debugging a Failure

Use this when a test fails or a bug is reported and you need Claude Code to trace it.

workflow — debugging 
# 1. Run the failing test and capture output
go test ./internal/auth/... -run TestValidateToken -v

# 2. Open Claude Code and describe the failure
The TestValidateToken test is failing with this output:
[paste the test output]
Read the test and the implementation, trace the failure, and tell me the root cause.
Do not fix anything yet — just explain what is wrong.

# 3. Confirm your understanding before asking for a fix
That analysis is correct. Now fix the bug in token.go line 47.
The fix should be minimal — do not refactor anything else.

# 4. Verify the fix works
Run TestValidateToken again and confirm it passes.
Tip: Always ask for the root cause analysis before asking for a fix. This forces Claude Code to show its reasoning — you can catch a wrong diagnosis before it produces a wrong fix. A one-sentence description of the bug that you agree with is worth more than jumping straight to a code change.

Pattern: Large-Scale Refactor

Renaming a function, changing an interface, or migrating a pattern across many files.

workflow — refactor 
# 1. Plan Mode first — understand the blast radius
#    Press Shift+Tab
I want to rename the getUserByID function to FindUserByID across
the entire codebase and update all callers. List every file that
needs to change and what change is needed in each.

# 2. Review the plan — are there files you did not expect?
# 3. Exit Plan Mode, execute
Execute the rename. Start with the definition in internal/store/user.go,
then update all callers. Run the build after each file to catch errors early.

# 4. Full test suite after all changes
Run go test ./... and fix any remaining compilation errors.

Pattern: Pre-PR Code Review

Use Claude Code as a final review pass before opening a pull request.

workflow — pre-PR review 
# Fresh session for clean context
claude

# Load only the diff — no extra context needed
Review this diff for: missing error handling, edge cases,
security issues, and deviations from the conventions in CLAUDE.md.
Do not suggest style preferences — only flag correctness issues.

# Paste the diff or reference the git diff
Here is the diff: @$(git diff main..HEAD)

# Or reference a specific file
Review the changes in @internal/handlers/payment.go
Advanced

Tips for Power Users

These patterns build on the fundamentals and are worth adopting once you are comfortable with the core workflow.

Advanced Usage Patterns

Use One-Shot Mode for Automation

The -p flag runs Claude Code headlessly — no REPL, just execute and exit. This is the integration point for CI/CD pipelines and shell scripts.

bash — one-shot in scripts 
# In a CI script — auto-generate release notes from git log
claude -p "Based on this git log, write a CHANGELOG entry for version $(cat VERSION):
$(git log --oneline v$(cat VERSION)..HEAD)"

# Run as a pre-commit hook — check for debug code
claude -p "Scan the staged diff for debug statements (fmt.Println, console.log,
debugger, TODO: remove). List each occurrence with file and line number." --output-format json

Use --output-format for Machine-Readable Output

When integrating Claude Code into scripts, --output-format json produces structured output instead of prose — easier to parse programmatically.

bash 
claude -p "List all exported functions in pkg/auth/ with their signatures" --output-format json | jq '.result'

Layer CLAUDE.md Files for Monorepos

In a monorepo with multiple services, place a root CLAUDE.md for shared conventions and per-service CLAUDE.md files for service-specific context. Claude Code loads both when working in a service subdirectory.

directory layout — monorepo CLAUDE.md hierarchy 
monorepo/
├── CLAUDE.md              ← shared: monorepo conventions, CI commands
├── services/
│   ├── auth/
│   │   └── CLAUDE.md      ← auth-specific: JWT patterns, token constraints
│   ├── payments/
│   │   └── CLAUDE.md      ← payments-specific: PCI rules, no logging card data
│   └── notifications/
│       └── CLAUDE.md      ← notifications-specific: provider patterns, retry policy

Custom Instructions via /compact

The /compact command accepts optional text instructions that tell Claude how to prioritize the summary. Use this to emphasize what matters for the next phase of work.

prompt 
/compact Focus on what was decided about the data model and what still needs to be implemented. Drop the debugging history.
Tip: Combine Plan Mode with the @ file reference to build structured technical proposals. Enter Plan Mode, describe the problem, and ask Claude Code to produce a structured ADR (Architecture Decision Record) referencing the relevant files. Review it, then exit Plan Mode to implement.

Parallel Subagents for Independent Tasks

If you have two tasks that do not share state — e.g., writing tests for module A and writing documentation for module B — open two terminal windows and run both as separate Claude Code sessions simultaneously. They will not interfere with each other.

Note: Parallel sessions should operate on non-overlapping files. If both sessions write to the same file, you will get conflicts. Use separate git branches and merge afterward, or partition the work so each session owns its files.
