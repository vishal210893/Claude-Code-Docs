Context Engineering

Chapter 02 Foundations

What is Context Engineering?

Context engineering is the practice of designing what information an LLM receives — when it receives it, in what form, and how much of it. It has superseded "prompt engineering" as the primary skill in building reliable AI agents.

inbox Where Context Comes From

Every LLM interaction draws from multiple context sources that accumulate in the context window simultaneously:

1
Developer context

Instructions, memory files, system prompts, and project specifications you have intentionally loaded. This is the context you design.

2
User context

The current message, follow-up questions, and any material the user pastes inline during the session.

3
Interaction history

Every prior turn in the conversation — both messages and model responses — accumulates and grows throughout the session.

4
Tool call results

The output of every file read, shell command, web fetch, and MCP tool call is added to the context. Long outputs compound quickly.

key Garbage In, Garbage Out

The quality of an agent's output is bounded by the quality of its context. Irrelevant, contradictory, or stale information in the context window directly degrades response quality — regardless of model capability. Context engineering is the act of keeping that window clean, relevant, and appropriately sized.

Problems

Context Degradation

Long-running agentic tasks accumulate context, and as the window fills, performance degrades in three distinct ways. Understanding these patterns is necessary to diagnose and prevent them.

warning The Three Degradation Types

dangerous

Context Poisoning

The model generates incorrect information that gets appended to history. Subsequent turns treat the hallucination as fact, compounding the error.

waves

Context Confusion

The model receives more information than is relevant to the current task. Signal-to-noise ratio drops, causing inconsistent behavior and missed instructions.

bolt

Context Clash

Contradictory instructions from different sources (system prompt vs user message vs memory file) create ambiguity the model resolves unpredictably.

The root cause of all three is the same: context growing beyond what is useful for the current task. The four strategies below are direct mitigations.

Solutions

The Four Strategies

Claude Code implements context engineering through four complementary strategies. Each addresses a different aspect of the context management problem.

StrategyProblem It SolvesMechanism
Persistent MemoryContext lost between sessionsStructured memory files loaded automatically
Intelligent RetrievalLoading irrelevant contextDynamic, conditional context injection

Strategy 1

Persistent Memory Architecture

Claude Code implements a three-tier memory hierarchy using CLAUDE.md files at different scopes. Each tier serves a distinct purpose and audience.

folder_open The Three Memory Tiers

Tier 1
Project Memory
CLAUDE.md in repo root
Tier 2
User Memory
~/.claude/CLAUDE.md
Tier 3
Dynamic Imports
@ syntax at runtime

Tier 1 — Project Memory

A CLAUDE.md file in your repository root is automatically loaded every time Claude Code starts in that directory. It is version-controlled and shared with the entire team.

This file is the canonical source of truth for:

  • Architecture overview — how the project is structured and why
  • Technology stack — frameworks, libraries, key dependencies
  • Coding conventions — style rules, naming patterns, anti-patterns to avoid
  • Build and test commands — how to run, build, lint, and test
  • Key file locations — where the important code lives
markdown — CLAUDE.md
# Project: Auth Service

## Architecture
Go microservice using chi router. PostgreSQL via pgx. JWT auth.
All handlers in internal/handlers/, business logic in internal/service/.

## Commands
- Build: `make build`
- Test:  `go test ./...`
- Lint:  `golangci-lint run`

## Conventions
- Errors always wrapped with context: `fmt.Errorf("handler: %w", err)`
- Table-driven tests with t.Run()
- Never use global state

Tier 2 — User Memory

Located at ~/.claude/CLAUDE.md, this file is personal and never committed. It encodes your preferences, working style, and cross-project instructions.

  • Preferred response verbosity
  • Personal coding conventions (e.g., "always use early returns")
  • Tools and aliases you use across projects
  • Tone and formatting preferences

Tier 3 — Dynamic Imports

Reference any file in a prompt using the @ prefix. The contents are injected inline at the point of reference:

prompt
Review the implementation against @docs/spec/auth.md and suggest improvements

Dynamic imports are ideal for context that is only relevant to specific tasks. Don't load every spec file by default — import them when needed.

lightbulb Specificity Wins

Claude Code traverses parent directories and inherits memory files. A CLAUDE.md inside internal/service/ will be loaded when working in that directory, in addition to the root one. Use this to give subsystems their own focused context without polluting the top-level file.

Strategy 2

Intelligent Context Retrieval

Rather than loading all available context upfront, Claude Code retrieves context conditionally — based on directory, file type, tool use, and explicit user signals.

search Discovery Mechanisms

Automatic Folder Traversal

When Claude Code opens or edits a file, it checks for CLAUDE.md files in that file's directory and all parent directories up to the project root. More specific context takes priority over general context.

Tool-Specific Context

Claude Code can inject context when specific tools are invoked. For example, a terminal tool use might load a COMMANDS.md reference automatically — without that reference being in the context when only reading files.

User Quick-Add with #

Type # in the Claude Code REPL to surface a file picker. Selected files are injected into the current context for that turn only — useful for ad-hoc reference without permanently adding to memory.

info Conditional Loading Pattern

A common pattern is to keep a general CLAUDE.md at root and store domain-specific memory in subdirectory files (e.g., frontend/CLAUDE.md, api/CLAUDE.md). Claude Code loads subsystem context only when actually working in that subsystem — reducing total context without losing coverage.

Strategy 3

Context Compression

When the conversation history grows too large to be useful, two commands let you reclaim context window space without losing the persistent memory you've invested in.

compress /clear vs /compact

/clear — Wipe Conversation

Removes all conversation history from the context window. Memory files (CLAUDE.md) are preserved and reloaded in the new context.

Use when: Starting a new task, context is polluted, or the conversation history is no longer relevant.

/compact — Compress Conversation

Summarizes the conversation into a compressed representation. Key decisions, code changes, and context are retained in condensed form.

Use when: You want to continue a long task but the history has grown unwieldy.

Starting Fresh Sessions

For major task switches, the most reliable approach is to exit Claude Code entirely and start a new session. Memory files are reloaded clean, without any prior turn residue.

bash
# Inside REPL — wipe conversation history, keep memory files
/clear

# Inside REPL — compress history to essential context
/compact

# Exit completely, start fresh next run
/exit
warning When to Intervene

Watch for these signals that context needs compression: responses becoming less precise, Claude Code "forgetting" constraints you set earlier in the session, or noticeably slower/more expensive responses. These indicate the context window is filling with low-signal content.

Strategy 4

Context Isolation via Subagents

The most powerful context engineering technique is using specialized subagents — each with a narrow, focused context window — rather than doing all work in a single accumulating session.

smart_toy The Specialization Principle

When the main Claude Code instance needs a code review, it spawns a dedicated code reviewer subagent with only the diff as context. That agent has never seen the debugging session, the failed test attempts, or the earlier design discussions. It reviews with fresh eyes.

Common subagent specializations:

  • Code Reviewer — reads only the diff, applies quality standards
  • Security Auditor — scans for vulnerabilities, no business logic context needed
  • Test Writer — receives the implementation and test framework context only
  • Documentation Writer — sees the public API surface, not the implementation details
  • Research Agent — performs web search and synthesis without modifying code
key Why Isolation Improves Quality

An agent with 200k tokens of mixed context — debugging, refactoring, configuration, planning — will produce lower quality specialized output than an agent with 10k tokens of precisely relevant context. Context isolation is a direct quality multiplier, not just an efficiency optimization.

Advanced

System Prompt Design

The system prompt (primarily your CLAUDE.md) is the most stable and influential piece of context. Designing it well requires avoiding two opposite failure modes.

balance The Goldilocks Zone

cancel Too Specific (Brittle)

Treating the LLM as a deterministic state machine. Exhaustive enumeration of every case. Hardcoded logic and decision trees. Maintenance nightmare — any uncovered case falls through to undefined behavior.

"If the user asks about X, do Y. If they ask about Z, do W. If both X and Z, do..."
cancel Too Vague (Unpredictable)

Insufficient signal. False assumptions about default behavior. Undefined boundaries lead to inconsistent results. "Do the right thing" without defining what right means for this context.

"Be helpful and write good code."

The Balanced Approach

Effective system prompts establish principles rather than procedures. LLMs excel at applying general rules to specific situations — lean into that capability instead of fighting it.

markdown — effective CLAUDE.md pattern
# Identity and scope (clear, not exhaustive)
You are working on the Auth Service — a Go microservice handling JWT authentication.

# Principles (not procedures)
- Prioritize correctness over cleverness
- When uncertain between two approaches, prefer the simpler one
- Security boundaries are hard constraints, not guidelines

# Boundaries (what NOT to do)
- Never modify the public API contract without flagging it
- Do not add dependencies not already in go.mod without asking

# Reasoning framework
For any non-trivial change: state what you are changing, why, and what could break.
lightbulb Principles Over Rules

A single well-stated principle handles more situations than ten specific rules. "Prefer the simpler implementation" handles thousands of micro-decisions. A rule like "use early returns when the condition has more than 2 branches" handles only one. Write principles; let the model apply them.

Foundations

The Context Window Explained

Before you can manage context effectively, you need to understand what it is and why it degrades. The context window is the single most important constraint in working with any LLM agent.

memory Context Window as Working Memory

Think of the context window as RAM for the AI. Just as a computer loads the currently-needed program data into RAM to work with it, Claude loads the currently-needed information into its context window to reason about it.

AnalogyContext Window EquivalentImplication
RAM capacityToken limit (e.g., 200k tokens)There is a hard upper bound — fill it and things break
Programs loaded in RAMFiles read + conversation historyEverything Claude has "seen" this session is in context
RAM getting fullContext window nearing limitPerformance degrades before the hard limit is hit
Closing programs to free RAM/clear or /compactReclaim context space without losing persistent memory
Hard disk (persistent storage)CLAUDE.md filesSurvives session end; reloaded into context on next start

The key insight from this analogy: not everything needs to be in RAM at once. A well-designed CLAUDE.md is a compact summary that fits in context efficiently. A badly managed session loads entire files and lengthy tool outputs that are no longer relevant, wasting context space on stale data.

Note: Tokens are roughly 3-4 characters of text. A 200k token context window sounds enormous — but a single large file read, a few test outputs, and a long conversation can fill it faster than you expect. Context engineering is the discipline of using that space deliberately.

data_usage What Consumes Context

Every item below adds tokens to the context window and is never removed until you run /clear or /compact:

chat

Conversation History

Every message you send and every response Claude gives is added to history. A 100-turn session has 100x the baseline context of a fresh start.

folder_open

File Reads

When Claude reads a file — even one you did not ask it to read — the full file contents enter the context window and stay there for the session.

terminal

Command Output

Running tests, builds, or linters generates verbose output. A failing test suite output can be thousands of tokens of stack traces and log lines.

description

CLAUDE.md Files

Memory files are loaded automatically. A bloated CLAUDE.md with 5,000 tokens of content uses 5,000 tokens of context before a single turn happens.

Strategy 1 — Deep Dive

CLAUDE.md Deep Dive

CLAUDE.md is the most powerful context management tool available. Investing 30 minutes to write a good one returns that investment on the first non-trivial session. Here is everything you need to know to write one effectively.

account_tree The Hierarchy: Global → Project → Local

Claude Code loads CLAUDE.md files in a specific precedence order. More specific files take priority over more general ones:

1
Global — ~/.claude/CLAUDE.md

Personal preferences that apply to every project on your machine. Never committed to version control. Contains your individual coding style, preferred response format, and cross-project tools.

2
Project Root — ./CLAUDE.md

Committed to the repository. Shared with the entire team. The primary source of project-specific context: architecture, stack, commands, conventions.

3
Subdirectory — ./internal/service/CLAUDE.md

Loaded automatically when Claude Code is working in that subdirectory. Use for subsystem-specific context that would clutter the root file.

Info: All three levels are loaded simultaneously and merged. If the global CLAUDE.md says "use early returns" and the project CLAUDE.md says "this service uses a specific error wrapping pattern", both instructions are active. There is no conflict unless they directly contradict each other.

checklist What to Put in CLAUDE.md

A well-structured project CLAUDE.md covers these sections. Not all are required for every project — include what is genuinely non-obvious.

SectionWhat to IncludePriority
Architecture OverviewService boundaries, key components, how the system fits togetherHigh
Tech StackLanguage version, frameworks, key libraries and why they were chosenHigh
Build & Test CommandsExact commands to build, test, lint, and run locallyHigh
Coding ConventionsNaming patterns, error handling style, anti-patterns to avoidHigh
Key File LocationsWhere the important code lives — handlers, models, config, testsMedium
Out-of-Bounds AreasFiles or patterns Claude should never modify without explicit instructionMedium
Known GotchasSubtle constraints, legacy decisions, non-obvious rulesMedium
PR/Commit ConventionsCommit message format, branch naming, PR description formatLow

article Full Example CLAUDE.md

This is a production-quality CLAUDE.md for a Go microservice. Use it as a template — replace the specifics with your own project's details.

markdown — CLAUDE.md (project root) 
# Auth Service

## Architecture
Go microservice for JWT authentication and user management.
Exposes a REST API on port 8080. Communicates with PostgreSQL for
user storage and Redis for token blacklisting.

Service layout:
- cmd/server/         — entry point, dependency wiring
- internal/handlers/  — HTTP handlers (chi router)
- internal/service/   — business logic (no HTTP concerns)
- internal/store/     — database access (pgx, no ORM)
- internal/middleware/ — auth, logging, recovery middleware
- pkg/jwt/            — JWT utility package (shared with other services)

## Tech Stack
- Go 1.23
- chi v5 (router)
- pgx v5 (PostgreSQL driver — never use database/sql)
- Redis 7 via go-redis v9
- JWT via golang-jwt/jwt v5
- Testify for unit tests, no test framework for integration tests

## Commands
- Build:     make build
- Test:      go test ./...
- Lint:      golangci-lint run
- Run local: make run (requires POSTGRES_URL and REDIS_URL in .env)
- Migrate:   make migrate-up / make migrate-down

## Coding Conventions
- All errors must be wrapped with context: fmt.Errorf("handler.CreateUser: %w", err)
- Use early returns for error handling — do not nest success paths
- Table-driven tests with t.Run() for all unit tests
- SQL queries use named parameters, never positional ($1, $2) for readability
- No global state. All dependencies injected through struct fields.
- HTTP handler functions must not contain business logic — delegate to service layer

## Key Constraints
- Never modify pkg/jwt/ — it is shared with other services and has its own release cycle
- Do not add new direct PostgreSQL queries in handlers/ — route through store/ layer
- Do not add dependencies to go.mod without asking — dependency changes need team review
- The /health endpoint must remain side-effect free (no DB calls)

## Known Gotchas
- The User struct has both ID (int64) and PublicID (uuid) — external APIs use PublicID only
- Token expiry is in UTC. The clock skew tolerance is 30 seconds (configured in pkg/jwt/config.go)
- Redis connection pool size is tuned for the production cluster — do not change pool settings
Tip: Keep CLAUDE.md under 500 lines. If it grows larger, split it: move subsystem-specific content into subdirectory CLAUDE.md files. The root file should be a quick orientation, not an exhaustive manual.
Strategy 3 — In Practice

Practical /compact Usage

The /compact command is a precision tool — it compresses your conversation history into a condensed summary without erasing your memory files or losing the thread of a long task.

save What Gets Preserved vs. Lost

✓ Preserved After /compact

preserved
CLAUDE.md memory files (reloaded)
Key decisions made in the session
Which files were edited and how
The current task goal and progress
Code that was written or changed
Errors encountered and how they were resolved

✗ Lost After /compact

lost
Verbatim conversation turns
Exact tool call inputs/outputs
Intermediate reasoning steps
Full file contents that were read
Specific error messages and stack traces
The chronological sequence of actions
Warning: After /compact, Claude Code works from a summary, not the original history. If you need to reference a specific intermediate state or exact error message from earlier in the session, note it down before compacting. The summary may omit details you consider important.

schedule When to Use /compact

SituationUse /compact?Alternative
Long task, still in progress, history getting unwieldyYes — /compact—
Task complete, starting something newNo/clear is cleaner
Context poisoning — bad info in historyNo/clear removes the bad info entirely
Approaching token limit warningYes — /compact firstNew session if task can be split
Debugging session, need to try different approachOptional/clear gives truly fresh eyes

Context Strategies in Practice

Theory is only useful if it changes behavior. These before/after examples show what good and bad context management look like in real usage.

compare_arrows Before and After

✗ Poor Context Management

session — problematic
// Monday morning: debug a login bug
// ... 40 turns of debugging ...
// ... fixes auth middleware ...

// Same session, Monday afternoon:
"Now can you add a new /metrics endpoint?"

// Claude still has 40 turns of auth
// debugging context. It will:
// - Reference auth code unnecessarily
// - Potentially apply auth patterns to
//   metrics that do not need them
// - Use more tokens for worse results

✓ Good Context Management

session — effective

// ... 40 turns of debugging ...
// ... fixes committed to git ...

/clear  // ← wipe auth debugging context

// Monday afternoon (clean context):
"Add a /metrics endpoint that exposes
Prometheus-format runtime stats.
See @spec/metrics.md for requirements."

// Claude now has: CLAUDE.md + spec only
// Clean, focused, no auth residue

✗ Overloaded CLAUDE.md

markdown
# Project Context
[2,000 lines of documentation,
every API endpoint listed,
full database schema,
all historical decisions,
meeting notes from last quarter,
onboarding guide for new engineers,
deployment runbook,

Loaded on every session whether relevant or not. Most of it noise for any given task.

✓ Focused CLAUDE.md with Dynamic Imports

markdown

Advanced — System Prompts

System Prompt Design Best Practices

The system prompt (via CLAUDE.md) is the highest-priority context in every session. These patterns make it consistently effective rather than just occasionally useful.

format_list_bulleted Concrete Patterns That Work

1. State Identity and Scope First

The first lines of CLAUDE.md should answer: what is this project and what is Claude's role in it? This anchors every subsequent decision.

markdown — effective opening
You are working on the Payments Service — a Go microservice that
processes credit card transactions. All changes must maintain PCI-DSS
compliance. When in doubt, choose the more conservative implementation.

2. Prefer Principles Over Procedures

A principle handles infinitely more situations than a procedure. Do not enumerate every case — state the rule that governs all cases.

✗ Procedural (Brittle)

markdown
If adding a function, wrap errors.
If editing a function, wrap errors.
If the function calls another function,
wrap those errors too.
If the function is in handlers/, prefix
with "handler.FunctionName:".
If it is in service/, prefix with
"service.FunctionName:"...

✓ Principled (Flexible)

markdown
Always wrap errors with the calling
function's package and name as context:
fmt.Errorf("package.FuncName: %w", err)
This applies everywhere in the codebase.

3. Make Boundaries Explicit

List what Claude should never do without explicit permission. Hard constraints prevent well-intentioned overreach.

markdown — effective constraints section
## Hard Constraints (never do without asking)
- Do not add entries to go.mod
- Do not modify database migration files
- Do not change any exported function signatures in pkg/
- Do not commit or push to git

4. Include a Reasoning Framework

Tell Claude how to approach ambiguity. A short reasoning protocol prevents guessing behavior on edge cases.

markdown — reasoning framework
## When Uncertain
- If a change touches more than 3 files, describe the plan before executing
- If two approaches are equally valid, choose the one that changes fewer files
- If a requirement is ambiguous, ask before implementing
Tip: After writing a CLAUDE.md, test it immediately with a fresh session. Ask Claude Code to summarize what it knows about the project. If the summary is accurate and complete, the context file is working. If it misses key constraints, add them and retest.
