Slash Commands

Slash commands are special instructions entered inside the Claude Code REPL. They control session behavior, context management, configuration, and navigation — without consuming LLM tokens or triggering a model response.

Note: Slash commands are typed directly into the Claude Code REPL prompt — they are not shell commands and cannot be combined with shell pipes or redirects. They take effect immediately without consuming API tokens.

Complete Slash Command Reference

CommandWhat It DoesWhen to Use
/init Analyze codebase and generate CLAUDE.md First time in a new project
/clear Remove all conversation history; reload memory files Starting a new task, or context has become noisy
/compact Compress conversation to essential context Long session, want to continue but reclaim space
/status Show session ID, context usage, active memory files Diagnosing context issues, noting session ID to resume
/model <name> Switch model mid-session Escalating to Opus for a hard problem, or downgrading for routine work
/models List available models and current selection Before switching — see what's available
/help Show all slash commands with descriptions Quick reference when you forget a command
/exit End the session gracefully Done for the day, or switching to a fresh context
/new Start a new conversation in the current session Quick context reset without leaving the REPL
/undo Revert the most recent file change Immediately after a bad edit
/doctor Run diagnostics — check API key, tool health, MCP servers Troubleshooting a broken setup or unexpected behavior
/memory View and manage loaded memory files and their content Auditing what context Claude Code has loaded from CLAUDE.md files
/config View and modify configuration settings interactively Adjusting behavior without editing settings.json by hand
/mcp Manage and authenticate MCP servers Checking MCP server status or re-authenticating a server
/resume <id> Resume a previous session by ID Continuing work from yesterday; or sharing a session with a teammate
/desktop Teleport the current session to the Claude Code desktop app Moving a CLI session to the desktop app for visual diff review
/teleport Resume a web session in the terminal Picking up a session started in the web interface in your local CLI
/permissions Show current allow/deny permission rules in effect Verifying what is pre-approved before running a risky workflow
Tip: Use /status at the start of long sessions to check how much context is already consumed. If you are above 50%, consider /compact to compress history before adding more large files — this keeps Claude focused and reduces cost.

Understanding /clear vs /compact vs /new

These three commands all reset context but in different ways. Choosing the right one for the situation avoids losing useful context unnecessarily:

CommandWhat is PreservedWhat is DiscardedBest When
/clear Memory files (CLAUDE.md), permissions config All conversation history, tool results, file contents Starting a completely different task in the same project
/compact Summarized version of the conversation Verbose output, intermediate steps, raw file reads Long task is still ongoing but context is getting full
/new Nothing — fresh conversation All history and loaded context Quick pivot to an unrelated question in the same REPL session

CLI Flags

Flags passed to the claude command control how Claude Code starts, what mode it enters, and how it processes output. These are especially useful in scripting and CI/CD contexts.

Core Flags

bash
# -p / --print: one-shot mode — run a task and print output, no REPL
claude -p "List all exported functions in internal/service/"

# --output-format: control output format in -p mode
claude -p "Review this PR" --output-format json

# --continue / -c: continue the most recent session
claude --continue

# --resume: resume a specific session by ID
claude --resume abc123def

# --model: specify model at startup
claude --model claude-opus-4-5

# --no-tools: disable all tools (pure LLM conversation)
claude --no-tools

# --version: print version and exit
claude --version

# --help: show all flags
claude --help

Output Format Options

FormatOutputBest For
textPlain text responseHuman reading, piping to other commands
jsonStructured JSON with result and metadataScripting, CI integration, parsing programmatically
stream-jsonNewline-delimited JSON events as they streamReal-time processing pipelines
bash — scripting example
# Use in a CI pipeline — exit code reflects success/failure
claude -p "Check if there are any obvious security issues in the changed files" \
  --output-format json | jq '.result'

Combining Flags for Automation

The real power of CLI flags comes from combining them. Here are the most useful combinations for different scenarios:

bash Common flag combinations 
# Run a one-shot task on a specific model, get JSON output
claude -p "Summarize the changes in this diff" \
  --model claude-haiku-4-5 \
  --output-format json

# Continue yesterday's session where you left off
claude --continue

# Resume a specific saved session (ID from /status)
claude --resume sess_abc123

# Pipe file content into a one-shot prompt
cat error.log | claude -p "Diagnose this error log and suggest a fix"

# Process multiple files with a loop
for f in src/**/*.go; do
  claude -p "Add godoc comments to all exported symbols in $f"
done
Warning: When piping into claude -p in scripts, be careful not to include sensitive data (API keys, passwords, user PII) in the stdin input. The content is sent to the Anthropic API. Review what you are piping before automating these patterns.

Keyboard Shortcuts

Inside the interactive REPL, keyboard shortcuts control mode switching, cancellation, and navigation. These are the ones that matter day-to-day.

⌨ REPL Shortcuts

ShortcutActionNotes
Shift+TabToggle Plan ModeSwitches between normal and read-only planning mode
Ctrl+CCancel current operationInterrupts a running tool or generation mid-stream
Ctrl+C Ctrl+CForce exitHard exit when the REPL is unresponsive
↑ / ↓Browse prompt historyNavigate previous inputs in the current session
TabAutocomplete file pathsWhen typing @ paths for file references
Ctrl+LClear terminal screenVisual cleanup — does not affect context
Ctrl+C is Safe

Ctrl+C interrupts the current generation or tool execution without exiting the REPL. If Claude Code is reading a large file you don't need, or executing a long command you want to stop, Ctrl+C cancels it cleanly. You can then provide a more focused instruction.

Plan Mode — The Most Underused Feature

Shift+Tab toggles Plan Mode. In Plan Mode, Claude Code will read files, explore the codebase, and think through an approach — but will not write any files or execute any commands. This is the safest way to validate Claude's understanding before letting it make changes.

Tip: Use Plan Mode at the start of any non-trivial task. Press Shift+Tab, describe what you want, and Claude will produce a detailed plan. Review the plan, correct any misunderstandings, then press Shift+Tab again to re-enter normal mode and say "proceed". This single habit prevents the majority of wasted iterations.

Security

Permission System

Claude Code prompts for permission before taking actions with side effects — writing files, running commands, making network requests. The permission system is how you maintain oversight of agentic behavior.

Interactive Permissions

When Claude Code wants to perform an action, it describes what it intends to do and asks for approval:

  • Allow once — permits this specific action
  • Allow always — adds the action/tool to your allowlist (persisted in settings.json)
  • Deny — blocks the action; Claude Code adjusts its approach

Configuration via settings.json

Pre-authorize known-safe actions in .claude/settings.json to reduce interruptions during automated or repetitive workflows:

json — .claude/settings.json
{
  "permissions": {
    "allow": [
      "Bash(npm run test:*)",
      "Bash(go test ./...)",
      "Bash(git diff:*)",
      "Bash(git log:*)",
      "Read(**)",
      "Edit(src/**)"
    ],
    "deny": [
      "Bash(git push:*)",
      "Bash(rm -rf:*)"
    ]
  }
}
Warning: Pre-authorizing broad permissions (e.g., Bash(*)) removes a critical safety layer. Scope your allowlist to the specific commands you know are safe. For production environments, always require explicit approval for any action that writes to shared infrastructure or external services.

Permission Pattern Syntax

Permission rules use a tool-name + optional glob pattern. Understanding the syntax helps you write precise rules that allow what you need without over-granting:

json Permission pattern examples 
{
  "permissions": {
    "allow": [
      "Read(**)",               // Allow reading any file
      "Edit(src/**)",           // Allow editing only under src/
      "Bash(go test ./...)",    // Allow this exact command only
      "Bash(git diff:*)",       // Allow git diff with any arguments
      "Bash(make *:*)",         // Allow any make target
      "WebSearch(*)"            // Allow all web searches
    ],
    "deny": [
      "Bash(git push:*)",       // Never push (require explicit approval)
      "Bash(rm -rf:*)",         // Never force-remove files
      "Edit(.env*)",            // Never touch .env files
      "Edit(*.yaml:prod*)"      // Never edit prod YAML
    ]
  }
}
Note: Deny rules always take precedence over allow rules. If a pattern matches both, the action is denied. This means you can safely grant broad Bash(*) and then deny specific dangerous patterns as exceptions.

The Three Permission Scopes

Permissions can be configured at three levels. Higher-priority scopes override lower-priority ones:

ScopeFile LocationApplies ToPriority
Local project.claude/settings.local.jsonOnly you, in this projectHighest
Project.claude/settings.jsonAll team members in this projectMedium
User global~/.claude/settings.jsonYou, across all projectsLowest

Todo Lists

Claude Code maintains an internal todo list when executing multi-step tasks. This list provides visibility into what is planned, in progress, and complete — and serves as a coordination mechanism for complex work.

How Todo Lists Work

When given a complex task, Claude Code automatically breaks it into steps and tracks them:

  • Steps are created at the start, based on an initial analysis of the task
  • Each step moves from pending → in progress → complete
  • Only one step is in progress at any time
  • The list updates in real-time as work proceeds
  • New steps are added if the task reveals additional requirements
Why This Matters

Todo lists prevent an LLM from "context-forgetting" mid-task — the written list serves as an external memory that survives context window pressure. They also give you visibility to intervene: if Claude Code is working on step 3 but you realize step 2 was wrong, you can correct it before the error propagates.

Triggering Todo Creation

You can explicitly ask Claude Code to plan before executing:

prompt Explicit planning request 
Before making any changes, create a todo list for migrating the auth system
from session-based to JWT. I'll review it before you start.

Intervening Mid-Task

Because the todo list is visible in real-time, you can safely interrupt a running task. If Claude Code has just completed step 2 but step 3 is about to do something wrong, press Ctrl+C to cancel the current operation, then provide a corrective instruction. Claude Code will update the todo list accordingly and continue from the corrected state.

Tip: If you want Claude Code to pause and check in after each step rather than running through the whole task automatically, include this in your prompt: "After completing each step, stop and wait for my approval before proceeding to the next one." This turns a fully autonomous workflow into a supervised one, which is safer for irreversible operations.

Todo List States Explained

StateVisualMeaning
Pending[ ]Planned but not yet started
In Progress[~]Currently being executed — only one at a time
Complete[x]Successfully finished
Blocked[!]Cannot proceed — waiting on input or a dependency
Customization

Custom Slash Commands

You can define your own slash commands by creating Markdown files in .claude/commands/. Custom commands are prompts that can be invoked by name with /your-command-name.

Creating a Custom Command

Custom commands are Markdown files placed in the .claude/commands/ directory (project-level) or ~/.claude/commands/ (personal, available in all projects). The filename becomes the command name, with hyphens allowed.

bash Directory structure 
.claude/
  commands/
    review-pr.md      # Invoked as /review-pr
    add-tests.md      # Invoked as /add-tests
    security-scan.md  # Invoked as /security-scan
    deploy.md         # Invoked as /deploy
Note: Commands in .claude/commands/ are project-scoped — they are committed to the repo and shared with your whole team. Commands in ~/.claude/commands/ are personal — they work across all your projects but are not shared. Use project-scoped commands for workflow standards; personal commands for your own shortcuts.

Command File Format and Frontmatter

Command files support optional YAML frontmatter to configure behavior. The body of the file is the prompt that will be executed when the command is invoked:

markdown .claude/commands/review-pr.md 
---
description: "Review the current git diff for issues before opening a PR"
allowed-tools: ["Read", "Bash"]
---

Review the current git diff for:
1. Logic correctness and edge cases
2. Error handling coverage
3. Test coverage for new code paths
4. Security implications (input validation, auth checks, secrets exposure)
5. Performance concerns for hot paths

For each issue found, cite the file and line number.
Summarize with: Critical | Warning | Suggestion counts.

Frontmatter Options

FieldTypePurpose
descriptionstringShort description shown in /help listing
allowed-toolsstring[]Restricts which tools this command can use (safety boundary)

Using $ARGUMENTS for Dynamic Commands

Use the $ARGUMENTS placeholder anywhere in your command file. When you invoke the command with additional text (/command some text here), that text is substituted in place of every $ARGUMENTS occurrence.

markdown .claude/commands/add-tests.md — uses $ARGUMENTS 
---
description: "Add unit tests for a specific function or file"
allowed-tools: ["Read", "Edit", "Bash"]
---

Add comprehensive unit tests for: $ARGUMENTS

Requirements:
- Use the existing test framework already in this project
- Follow the same file naming and organization patterns as existing tests
- Cover: happy path, edge cases, error conditions
- Use table-driven test patterns where appropriate
- Do not modify the implementation — only add tests
- Run the tests after writing them to verify they pass

Invoked as: /add-tests internal/cache/cache.go — the file path is inserted where $ARGUMENTS appears.

Practical Example: /deploy Command

markdown .claude/commands/deploy.md 
---
description: "Pre-deployment checklist and deploy to environment: $ARGUMENTS"
allowed-tools: ["Read", "Bash"]
---

Prepare to deploy to environment: $ARGUMENTS

Before deploying, verify:
1. Run the full test suite and confirm it passes
2. Check for any uncommitted changes (git status)
3. Confirm the current branch is the expected release branch
4. Review the diff from the last tag to HEAD (git log --oneline)
5. Check for any TODO or FIXME comments in recently changed files

If all checks pass, print a deployment summary and ask for confirmation before proceeding.
If any check fails, stop and report the issue clearly.

Invoked as: /deploy staging or /deploy production

Practical Example: /security-scan Command

markdown .claude/commands/security-scan.md 
---
description: "Scan the codebase for common security vulnerabilities"
allowed-tools: ["Read", "Bash"]
---

Perform a security scan of the codebase. Check for:

**Secrets and credentials**
- Hardcoded API keys, tokens, passwords
- Credentials in config files or comments

**Input validation**
- Unsanitized user input passed to shell commands, SQL, or file paths
- Missing validation on API endpoints

**Authentication and authorization**
- Endpoints accessible without authentication
- Missing permission checks on sensitive operations

**Dependencies**
- Run `npm audit` or `go list -m all` to identify known CVEs

Report each finding with: file path, line number, severity (Critical/High/Medium), and a recommended fix.

Command Design: Good vs Bad Practices

Well-Designed Command
---
description: "Review PR diff for correctness and security"
allowed-tools: ["Read", "Bash"]
---

Review the git diff between HEAD and main.

Focus on:

  1. Correctness — logic errors, edge cases
  2. Security — input validation, auth
  3. Error handling — all paths covered
  4. Test coverage — new code is tested

For each issue: cite file:line, classify as CRITICAL | WARNING | SUGGESTION.

End with a pass/fail recommendation.

Poorly-Designed Command

Review the code and tell me if there are any problems.

Also maybe check for bugs.

Fix anything that looks wrong.

Tip: The best custom commands encode your team's specific standards — your error wrapping conventions, your required test patterns, your security requirements. A generic "review the code" command is less valuable than one that checks against your actual team norms. Spend time making your commands specific to your project.

Integrating Custom Commands with Hooks

Commands and hooks are complementary. A common pattern: use a PostToolUse hook to automatically trigger a command whenever Claude Code saves a file. This creates a review-on-save loop:

json .claude/settings.json — hook triggering after edits 
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "echo 'File modified: running linter' && golangci-lint run --fast 2>&1 | head -20"
          }
        ]
      }
    ]
  }
}
Automation

Hooks

Hooks are shell commands that execute automatically on Claude Code lifecycle events. They are the primary mechanism for integrating Claude Code into larger automation workflows.

Available Hook Events

EventFires WhenCommon Use
PreToolUseBefore any tool is invokedAudit logging, safety checks, dry-run mode
PostToolUseAfter a tool completesNotifications, auto-formatting, test triggering
NotificationWhen Claude Code sends a notificationDesktop alerts, Slack messages, audit trails
StopWhen a session endsPlaying a sound, sending a summary, cleanup
json — .claude/settings.json
{
  "hooks": {
    "Stop": [
      {
        "matcher": "",
        "hooks": [
          { "type": "command", "command": "afplay ~/sounds/done.wav" }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          { "type": "command", "command": "echo \"$(date): file modified\" >> ~/.claude/audit.log" }
        ]
      }
    ]
  }
}
Real-World Hook Uses

The most common production hook patterns: running gofmt or prettier after every file edit, sending a Slack notification when a long-running task completes, logging all tool invocations to an audit file for compliance, and auto-running tests after any source file change.

Hook Matchers in Detail

The matcher field in a hook configuration filters which tool invocations trigger the hook. It uses simple string matching against the tool name:

json Hook matcher patterns 
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "",            // Empty string matches ALL tools
        "hooks": [{ "type": "command", "command": "echo triggered" }]
      },
      {
        "matcher": "Write",       // Matches only the Write tool
        "hooks": [{ "type": "command", "command": "prettier --write $TOOL_INPUT_PATH" }]
      },
      {
        "matcher": "Write|Edit",  // Matches Write OR Edit (pipe = OR)
        "hooks": [{ "type": "command", "command": "git add -p" }]
      },
      {
        "matcher": "Bash",        // Matches the Bash tool
        "hooks": [{ "type": "command", "command": "echo \"$(date): bash\" >> audit.log" }]
      }
    ]
  }
}

Hook Environment Variables

When a hook command executes, Claude Code injects several environment variables you can use in your hook scripts:

VariableAvailable InContains
TOOL_NAMEPreToolUse, PostToolUseName of the tool being invoked (e.g., Write, Bash)
TOOL_INPUT_PATHPostToolUse (file tools)Path of the file that was written or edited
TOOL_EXIT_CODEPostToolUse (Bash)Exit code of the bash command that ran
SESSION_IDAll hooksCurrent session ID for correlation in logs

Complete Hook Examples

json Auto-format on save (Go + TypeScript) 
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "if [[ \"$TOOL_INPUT_PATH\" == *.go ]]; then gofmt -w \"$TOOL_INPUT_PATH\"; elif [[ \"$TOOL_INPUT_PATH\" == *.ts || \"$TOOL_INPUT_PATH\" == *.tsx ]]; then npx prettier --write \"$TOOL_INPUT_PATH\"; fi"
          }
        ]
      }
    ]
  }
}
json Audit log — record every tool invocation 
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "echo \"$(date -u +%Y-%m-%dT%H:%M:%SZ) [$SESSION_ID] $TOOL_NAME\" >> ~/.claude/audit.log"
          }
        ]
      }
    ]
  }
}
json Desktop notification on session end 
{
  "hooks": {
    "Stop": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "osascript -e 'display notification \"Claude Code session complete\" with title \"Claude Code\"'"
          }
        ]
      }
    ]
  }
}
Warning: Hook commands run with the same permissions as your shell. A malicious command could be injected via a PreToolUse hook if the hook reads from an untrusted source. Keep hook scripts simple and side-effect-free. Avoid running hooks that themselves invoke Claude Code — this creates infinite loops.
