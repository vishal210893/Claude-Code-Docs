Chapter 05 Foundations

What is MCP?

The Model Context Protocol (MCP) is an open standard that defines how AI models communicate with external tools and data sources. Before MCP, every tool integration was bespoke — each agent had its own proprietary protocol. MCP proposes one universal interface that any model and any tool can speak.

The Core Problem MCP Solves

Prior to MCP, integrating an AI agent with, say, GitHub required writing custom code to translate between the agent's expectations and GitHub's API. The same was true for every other tool: databases, filesystems, Slack, browsers, monitoring systems.

MCP standardizes this by defining:

  • How tools describe themselves — JSON Schema-based tool definitions
  • How agents invoke tools — a standard request/response protocol
  • How results are returned — structured responses the model can reason over
  • Transport options — stdio (local processes), SSE (remote servers), HTTP
The Value Proposition

One GitHub MCP server works with Claude Code, GPT-4, Gemini, and any other MCP-compatible agent. One investment in a well-designed MCP server creates a reusable integration asset. The standardization goal is valid and compelling — the current implementation has specific limitations covered in the Pitfalls section.

MCP vs Direct Tool Calls — What's the Difference?

Claude Code has built-in tools (Read, Edit, Bash, WebSearch). MCP servers add additional tools on top of these. Think of built-in tools as the standard library, and MCP servers as installable packages that extend what Claude Code can do.

CapabilityBuilt-in ToolsMCP Servers
Read/write local filesYes (Read, Edit, Write)Via filesystem MCP (extends scope)
Run shell commandsYes (Bash)Not typically — Bash handles this
GitHub API accessNoYes (GitHub MCP)
Browser automationNoYes (Playwright MCP)
Database queriesNoYes (Postgres MCP, etc.)
Persistent memoryNoYes (Memory MCP)
Web scrapingLimited (WebSearch)Yes (Playwright, Fetch MCP)
How It Works

Architecture

MCP uses a client-server model. Claude Code is the MCP client. External tools run as MCP servers. The model sends requests to servers and receives structured results.

Request Flow

Transport Mechanisms

TransportHow It WorksBest For
stdioClaude Code spawns a local process and communicates via stdin/stdoutLocal tools, filesystem access, system integrations
SSERemote server pushes events over HTTP Server-Sent EventsRemote APIs, shared team servers, cloud tools
HTTPStandard request/response over HTTPSimple integrations, REST API wrappers

How Tool Discovery Works

When Claude Code starts and MCP servers are configured, it performs a tool discovery handshake. Here is what happens step by step:

StepWhat Happens
1. Server StartClaude Code launches the MCP server process (stdio) or connects to the remote URL (SSE/HTTP)
2. Capability NegotiationClient and server exchange supported MCP protocol versions and capability flags
3. Tool List RequestClaude Code sends a tools/list request to enumerate all available tools
4. Schema InjectionAll tool schemas from all configured servers are loaded into the active context window
5. ReadyClaude Code can now invoke any discovered tool by sending a tools/call request with arguments
Note: Step 4 is the source of the context pollution problem described in the Pitfalls section. All tool schemas are injected at startup, not on demand. A server with 40 tools injects 40 schemas even if you only use 2 of them. This is a current limitation of the MCP protocol.
Setup

Connecting MCP Servers

Claude Code provides first-class commands for installing, configuring, and managing MCP servers at both user and project scope.

Installation Commands

bash
# Add from npm package (stdio is the default transport)
claude mcp add -s user github -- npx @anthropic-ai/mcp-server-github

# Add from a local path
claude mcp add -s project myserver -- /path/to/my-mcp-server

# Add with environment variables (for API keys)
claude mcp add -e GITHUB_TOKEN=$GITHUB_TOKEN -s user github -- npx @anthropic-ai/mcp-server-github

# List installed servers
claude mcp list

# Remove a server
claude mcp remove github

Manual Configuration

MCP server configurations are stored in .claude/settings.json (project) or ~/.claude/settings.json (user). You can edit them directly:

json settings.json — multiple MCP servers 
{
  "mcpServers": {
    "playwright": {
      "type": "stdio",
      "command": "npx",
      "args": ["@playwright/mcp@latest"]
    },
    "github": {
      "type": "stdio",
      "command": "npx",
      "args": ["@anthropic-ai/mcp-server-github"],
      "env": { "GITHUB_TOKEN": "${GITHUB_TOKEN}" }
    },
    "postgres": {
      "type": "stdio",
      "command": "npx",
      "args": ["@anthropic-ai/mcp-server-postgres"],
      "env": { "DATABASE_URL": "${DATABASE_URL}" }
    },
    "remote-api": {
      "type": "sse",
      "url": "https://mcp.example.com/sse",
      "headers": { "Authorization": "Bearer ${API_TOKEN}" }
    }
  }
}

Configuration Scopes

MCP servers can be configured at different scopes, controlling who has access to which servers:

ScopeFileWho Has AccessUse Case
User global~/.claude/settings.jsonYou, in all projectsPersonal tools: your GitHub token, personal Postgres
Project shared.claude/settings.jsonAll team membersTeam tools: shared staging DB, company GitHub server
Project local.claude/settings.local.jsonOnly you, in this projectDeveloper overrides: local DB instance, personal test tokens
Tip: Commit .claude/settings.json (project shared) to give your whole team access to the same MCP servers with the same configuration. Add .claude/settings.local.json to .gitignore so individual developer overrides (personal tokens, local URLs) are never committed.

Step-by-Step: Setting Up the Filesystem MCP

The filesystem MCP gives Claude Code access to directories outside the current working directory. Useful for reading documentation, configuration files, or shared resources in other locations.

bash Install filesystem MCP 
# Step 1: Install the package globally
npm install -g @anthropic-ai/mcp-server-filesystem

# Step 2: Add to user-level settings
claude mcp add -s user filesystem -- mcp-server-filesystem /Users/me/Documents /Users/me/projects

# Step 3: Verify it is listed
claude mcp list

# Step 4: Test in REPL
# Type this in Claude Code:
# "List all Markdown files in /Users/me/Documents/notes/"

Step-by-Step: Setting Up the Playwright MCP

Playwright MCP gives Claude Code the ability to control a browser — navigate pages, click elements, fill forms, take screenshots, and extract content.

bash Install Playwright MCP 
# Step 1: No separate install needed — npx handles it
# Add directly to settings.json:
claude mcp add -s user playwright -- npx @playwright/mcp@latest

# Step 2: Install browser binaries (one-time)
npx playwright install chromium

# Step 3: Test in REPL:
# "Navigate to docs.anthropic.com and take a screenshot"

Step-by-Step: Setting Up the GitHub MCP

bash Install GitHub MCP 
# Step 1: Create a GitHub Personal Access Token
# Go to: GitHub → Settings → Developer settings → Personal access tokens → Fine-grained
# Required scopes: repo, read:org, workflow (if using Actions)

# Step 2: Add the token to your shell profile
echo 'export GITHUB_TOKEN="ghp_your_token_here"' >> ~/.zshrc
source ~/.zshrc

# Step 3: Install the MCP server
claude mcp add -e GITHUB_TOKEN="${GITHUB_TOKEN}" -s user github -- npx -y @anthropic-ai/mcp-server-github

# Step 4: Verify connection in REPL:
# "List my 5 most recently updated repositories"

Internals

Tool Schemas

Every MCP tool is described by a JSON Schema definition. Claude Code loads these schemas at startup — they define what tools exist and what arguments they accept. The quality of these schemas directly affects how well the model uses the tools.

Schema Structure

json — example tool definition
{
  "name": "github_create_pull_request",
  "description": "Create a pull request in a GitHub repository",
  "inputSchema": {
    "type": "object",
    "properties": {
      "owner":  { "type": "string", "description": "Repository owner" },
      "repo":   { "type": "string", "description": "Repository name" },
      "title":  { "type": "string", "description": "PR title" },
      "body":   { "type": "string", "description": "PR body in Markdown" },
      "head":   { "type": "string", "description": "Source branch" },
      "base":   { "type": "string", "description": "Target branch" }
    },
    "required": ["owner", "repo", "title", "head", "base"]
  }
}

The model reads these schemas to understand what tools are available and how to use them. Better descriptions mean better tool usage — vague or minimal descriptions lead to misuse or omission.

What Makes a Good Tool Schema?

The difference between a tool that Claude uses correctly 95% of the time versus 60% of the time is almost entirely in the quality of its schema description. Here is what separates good from bad:

✓ Good Schema Description

json
{
  "name": "search_issues",
  "description": "Search GitHub issues and PRs using GitHub's search syntax. Use this when you need to find specific issues by keyword, label, state, or author. For listing all open issues, use list_issues instead. Example: q='is:open label:bug' searches for open bugs.",
  "inputSchema": {
    "properties": {
      "q": {
        "type": "string",
        "description": "GitHub search query. Supports: is:open, is:closed, label:name, author:username, created:>2024-01-01. Required."
      },
      "per_page": {
        "type": "integer",
        "description": "Results per page, 1-100. Default: 30.",
        "default": 30
      }
    }
  }
}

✗ Poor Schema Description

json
{
  "name": "search_issues",
  "description": "Search issues",
  "inputSchema": {
    "properties": {
      "q": {
        "type": "string"
      },
      "per_page": {
        "type": "integer"
      }
    }
  }
}

Building a Simple MCP Server — Conceptual Walkthrough

Understanding what an MCP server does internally helps you evaluate third-party servers and debug issues. An MCP server is a process (any language) that:

  • Implements the MCP protocol — responds to initialize, tools/list, and tools/call requests
  • Declares its tools — returns JSON Schema definitions in response to tools/list
  • Executes tool calls — when Claude sends tools/call with a tool name and arguments, it performs the actual work and returns a structured result
  • Handles errors gracefully — returns structured error responses rather than crashing
typescript Minimal MCP server structure (conceptual) 
import { Server } from "@modelcontextprotocol/sdk/server/index.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";

const server = new Server(
  { name: "my-server", version: "1.0.0" },
  { capabilities: { tools: {} } }
);

server.setRequestHandler("tools/list", async () => ({
  tools: [
    {
      name: "get_weather",
      description: "Get current weather for a city. Returns temperature and conditions.",
      inputSchema: {
        type: "object",
        properties: {
          city: { type: "string", description: "City name, e.g. 'London'" }
        },
        required: ["city"]
      }
    }
  ]
}));

server.setRequestHandler("tools/call", async (request) => {
  if (request.params.name === "get_weather") {
    const { city } = request.params.arguments;
    const data = await fetchWeatherAPI(city);
    return { content: [{ type: "text", text: JSON.stringify(data) }] };
  }
  throw new Error(`Unknown tool: ${request.params.name}`);
});

const transport = new StdioServerTransport();
await server.connect(transport);
Note: You do not typically need to write MCP servers from scratch. The community ecosystem already covers most common tools. Building custom servers makes sense when you need to expose proprietary internal APIs, connect to internal databases not exposed to the internet, or create highly specialized tools for your specific workflow.
Critical Reading

Known Pitfalls

MCP's current implementation has structural limitations that matter at production scale. Being aware of these prevents common failure modes in agentic systems.

Context Pollution

MCP servers load all their tool schemas into the context window at startup — regardless of which tools will actually be used. A server with 50 tools injects 50 schema definitions even if only 2 are needed for the task.

At scale, this causes:

  • Token waste — every request pays for irrelevant tool definitions
  • Degraded selection — the model must search a larger tool space, increasing mis-selection probability
  • Context pressure — tool schemas compete with actual task content for context window space

Mitigation: Install only the servers you need. For large servers, prefer minimal servers that expose only what's needed for the current workflow.

Round-Trip Inefficiency

Complex tasks that require multiple tool calls execute as a sequence of individual round trips: model → tool → model → tool → model. Each round trip is a full API call with full context.

A task requiring 8 tool calls requires 8 complete round trips. As context grows with each result, later calls become more expensive and slower than earlier ones.

Mitigation: Design workflows to minimize sequential tool calls. Where possible, batch operations or use subagents to parallelize independent tool sequences.

Shallow Tool Definitions

JSON Schema captures parameter types and names — but not usage patterns, failure modes, or best practices. A schema tells the model what a tool accepts, not how to use it effectively.

The gap between "schema describes the tool" and "model uses the tool well" is bridged by:

  • Rich description fields with usage examples
  • Explicit notes about common mistakes in parameter values
  • Tool-specific context in your CLAUDE.md files
Warning: When writing custom MCP servers, the description field in your tool schema is as important as the implementation. Include: what the tool does, when to use it vs alternatives, what a correct call looks like, and common parameter mistakes to avoid. Good descriptions measurably improve tool usage accuracy.

Security Considerations for MCP

MCP servers execute with the permissions of the process they run in. A compromised or malicious MCP server can do significant damage. Before trusting any MCP server, evaluate:

QuestionWhy It Matters
Is the server open source?Can you audit what it actually does with the data and credentials you give it?
What credentials does it request?Minimal-scope tokens are safer. Does it need write access, or would read-only suffice?
Is it maintained?Unmaintained MCP servers may have security vulnerabilities in dependencies.
Does it log data?Some MCP servers log all requests to external analytics. Check the privacy policy.
What happens on errors?Does it expose internal error messages, stack traces, or connection strings in error responses?
Warning: Prompt injection via MCP is a real risk. If an MCP server reads content from the web (e.g., a web scraping MCP), that content could contain instructions designed to hijack Claude's behavior. This is called an "indirect prompt injection attack." Be especially cautious with MCPs that fetch arbitrary external content and feed it back to the model.
Guidance

MCP Best Practices

Production MCP Checklist

  • Install only what you need. Each installed server adds context overhead. Prefer targeted servers over comprehensive ones.
  • Use project scope for team tools. Commit your settings.json MCP configuration so every team member gets consistent tool access.
  • Secure API tokens via environment variables. Never hardcode credentials in settings files. Use ${ENV_VAR} syntax in the configuration.
  • Write tool-specific CLAUDE.md context. Load @memory/github/CLAUDE.md when doing GitHub work, with conventions specific to your repo and team workflow.
  • Test tool definitions independently. Before relying on an MCP server in production workflows, verify that the model selects and invokes the tools correctly with representative test prompts.
  • Monitor token usage. MCP tool calls are some of the highest-token operations in Claude Code. Audit which tools get called most and whether the results justify the cost.

Troubleshooting MCP — Common Issues and Fixes

SymptomLikely CauseFix
MCP tools not appearing in Claude Code Server failed to start or handshake failed Run claude mcp list to check status. Check that the command and args in settings.json are correct. Run the command manually in your terminal to see startup errors.
Authentication errors from the MCP server Token not available in environment Verify echo $GITHUB_TOKEN returns a value. Ensure the variable is exported, not just set. Restart Claude Code after sourcing your shell profile.
Claude Code invokes wrong tool Ambiguous or poor tool descriptions Add context to your CLAUDE.md explaining which tools to prefer for which tasks. Improve the tool's description field.
MCP server crashes mid-session Unhandled error in the server process Check the server's stderr output. For stdio servers, add 2>&1 to your command to capture stderr. Report to the server's maintainers.
Tools work once then stop Token expired or rate limit hit Refresh the token. Add rate limit handling context in CLAUDE.md: "If GitHub returns 429, wait 60 seconds before retrying."
SSE server connection drops Network timeout or server restart Claude Code will attempt to reconnect. If persistent, check the server's uptime and any firewall rules blocking the connection.
