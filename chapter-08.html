Subagents

Claude Code can spawn child agents to handle focused subtasks in parallel — each with its own isolated context window. This chapter covers how subagents work, how context flows between them, how to prompt them effectively, and advanced patterns like meta prompting and infinite subagent orchestration.

Chapter 08 Fundamentals

What Are Subagents?

Subagents are child Claude Code instances spawned by a main agent to handle specific, bounded subtasks. The main agent acts as an orchestrator — decomposing work, delegating to subagents, and synthesizing results — while each subagent operates in its own isolated context window.

The Orchestrator Pattern

When a task is large enough to risk context window saturation, or when subtasks can be parallelized, the main Claude Code agent spawns subagents. Each subagent receives a targeted task description and a slice of relevant context — not the full conversation history. This keeps each subagent focused and prevents the context pollution that degrades output quality on long-running sessions.

lock
Context Isolation
Each subagent has its own context window. Irrelevant history from the main session does not pollute subagent reasoning.
bolt
Parallel Execution
Multiple subagents run concurrently. A task that would take 20 minutes sequentially can complete in under 5 minutes with parallel subagents.
gps_fixed
Specialized Focus
Each subagent is given a narrow scope — a single file, a single concern, a single question. Narrow scope produces sharper results.
construction
Hierarchical Delegation
The main agent coordinates and synthesizes. Subagents execute. This mirrors how engineering teams actually work: lead designs, engineers implement.
info When Claude Code Uses Subagents Automatically

Claude Code will autonomously spawn subagents when a task involves multiple independent workstreams, when the codebase is large enough to require partitioned analysis, or when you instruct it to work in parallel. You can also explicitly request subagent delegation in your prompts — the model responds to both implicit task structure and explicit instructions.

Mechanics

Spawning Subagents

Claude Code uses the Task tool internally to spawn subagents. You can trigger this behavior by structuring your prompts to describe parallel workstreams — Claude Code will map each workstream to a subagent invocation automatically.

Instructing Claude to Use Subagents

You do not need to invoke any special syntax. Describe the work in terms of parallel, focused tasks and Claude Code will decompose it into subagent calls:

prompt — explicit parallel subagents
Research how our API authentication works, then:
- Spawn a subagent to audit the auth middleware
- Spawn another subagent to check JWT validation
- Spawn a third subagent to review token refresh logic
Synthesize findings into a security report.

You can also describe the parallelism more naturally and Claude Code will infer the subagent structure:

prompt — natural parallelism
Audit the entire authentication system in parallel. Cover middleware,
JWT handling, token refresh, session management, and OAuth callbacks.
Each area should be reviewed independently, then combine all findings
into a single prioritized risk report.
info How the Task Tool Works

When you instruct Claude to "spawn agents" or "use parallel agents", it invokes the Task tool to create subagents with dedicated context windows. Each Task call launches an independent agent that executes its assignment and returns a result to the main agent — which then continues with the synthesized output.

Controlling Subagent Count

You can give explicit guidance on parallelism to match your token budget or time constraints:

prompt — bounded parallelism
Review all Go files in internal/handlers/ for error handling issues.
Use up to 4 parallel subagents — assign each a portion of the files.
Report back all findings with file path, line number, and severity.
Architecture

Subagents Context Flow

Understanding how context flows between the main agent and subagents is essential for writing effective subagent prompts. Context is not automatically shared — the main agent controls what each subagent sees.

Flow Diagram

What Flows and What Doesn't

DirectionWhat is passedWhat is NOT passed
Main → Subagent Task description, relevant file content, explicit context the main agent includes Full conversation history, other subagents' tasks, unrelated main-agent context
Subagent → Main Task results, findings, generated code or content The subagent's internal reasoning steps or tool call history
Subagent ↔ Subagent Nothing — subagents are fully isolated from each other Everything
warning Subagents Do Not Share Context

Subagents do not share context with each other. If subagent 2 needs to know what subagent 1 found, the main agent must explicitly pass that information in the task description for subagent 2. Plan your task decomposition so that truly independent tasks go to separate subagents — interdependent tasks should either be handled by a single subagent or sequenced with the main agent relaying results.

Passing Cross-Subagent Context Explicitly

prompt — relaying context between subagents
Step 1: Spawn a subagent to map all authentication entry points in the
codebase. Return the file list and function signatures.

Step 2: Using the entry points found in step 1, spawn a subagent for
each entry point to audit its input validation. Pass the specific
file and function from step 1 into each subagent's task description.
Quality

Prompting Subagents Effectively

The quality of subagent output is directly proportional to the specificity of the task description. Vague tasks produce vague results. Narrow, well-scoped tasks with explicit output format requirements produce actionable findings.

Scope and Output Format

Two factors dominate subagent output quality: how narrow the scope is, and how explicitly the output format is specified. Subagents should not need to infer what you want — tell them exactly.

Poor subagent task description
  • Check if our code is secure
  • Review the auth module
  • Look for bugs in the handlers
  • Analyze performance
Effective subagent task description
  • Audit src/auth/jwt.go lines 45–120 for weak signing algorithms, missing expiry validation, and improper error handling. Return findings as JSON: {file, line, severity, description}
  • Read middleware/auth.go. For each middleware function, verify it calls next() only after successful auth. List any function where unauthenticated requests can pass through.
  • Review internal/handlers/user.go for SQL injection risks. Check all database queries. Return file:line for each risk found.
  • Profile the top 3 most-called functions in pkg/cache/. Identify any O(n²) or worse operations. Suggest the specific change to improve each one.

Output Format Specification

Always specify the output format explicitly. Structured output from subagents makes the main agent's synthesis step reliable and consistent:

prompt — structured output requirement
Audit src/auth/jwt.go for:
1. Weak signing algorithms (HS256 without rotation, none algorithm)
2. Missing expiry validation
3. Improper error handling that leaks token structure

Return findings as a JSON array with this shape:
[
  {
    "file": "src/auth/jwt.go",
    "line": 87,
    "severity": "CRITICAL",  // CRITICAL | HIGH | MEDIUM | LOW
    "category": "weak-algorithm",
    "description": "HS256 used without key rotation policy"
  }
]

If no issues are found, return an empty array.
lightbulb Scope by File Range, Not by Concept

The most reliable way to scope subagents is by specific file and line range — not by vague concept like "the auth system". Telling a subagent exactly which file and which lines to examine eliminates ambiguity about what is in scope, produces consistent results, and allows you to predict exactly how many subagents are needed to cover a codebase.

Advanced

Meta Prompting

Meta prompting is the practice of using Claude to write prompts for other Claude instances. The main agent generates the task descriptions that subagents will execute — enabling systematic, high-quality task decomposition at scale without you having to manually craft every subagent prompt.

The Meta Prompting Pattern

1
User provides the high-level goal
You describe the overall objective — "security audit the entire authentication system" — without enumerating every file or subtask.
2
Main agent explores and decomposes
The main agent reads the codebase, identifies all relevant files and modules, and generates a list of specific subagent task descriptions — one per work unit.
3
Main agent writes specialized prompts
For each work unit, the main agent writes a detailed, scoped task description — file path, line range, specific checks, output format. These become the subagent prompts.
4
Subagents execute
Each subagent receives its generated task description and executes against it. Narrow scope + explicit output format = consistent, usable results.
5
Main agent aggregates
All subagent results are returned to the main agent, which synthesizes them into the final output — a structured report, a summary, or a set of changes.

Meta Prompting in Practice

Instruct Claude to generate subagent task descriptions as its first step, then execute them:

prompt — meta prompting instruction
First, explore the repository structure of internal/handlers/.
For each Go file you find, generate a subagent task description that:
- Names the specific file
- Lists exactly what to check (error handling, nil checks, input validation)
- Specifies JSON output format: {file, line, issue, severity}

Then spawn those subagents in parallel and aggregate all findings
into a single markdown security report sorted by severity.
info Meta Prompting vs Manual Decomposition

Writing subagent prompts manually is practical for 3–5 subagents. Beyond that, meta prompting scales better — the main agent knows the codebase structure and can generate precise, consistent task descriptions for dozens of subagents without you having to enumerate every file. Meta prompting is particularly effective for large codebases where the full scope of work is not known upfront.

Scale

Infinite Subagents Pattern

The infinite subagents pattern extends meta prompting to its logical limit: the main agent dynamically spawns as many subagents as needed based on what it discovers in the codebase. Each completed subagent informs whether additional subagents are required.

How It Works

1
Main agent analyzes repository structure
The main agent traverses the codebase — listing files, identifying modules, reading package-level documentation — to understand the full scope of the task.
2
Generates N task descriptions
Based on the discovery step, the main agent generates one task description per file, module, or logical work unit. This list may be 10 items for a small service or 200+ items for a large monorepo.
3
Spawns N subagents in parallel
All generated task descriptions are dispatched as parallel subagent invocations. Execution happens concurrently — the total time is bounded by the slowest subagent, not the sum of all.
4
Each subagent executes and reports
Subagents complete their narrowly scoped tasks and return structured results. No subagent is aware of the others.
5
Main aggregates all results
The main agent collects all subagent outputs, deduplicates findings, prioritizes by severity or impact, and produces the final deliverable.

Best Use Cases

operations across many files
Parallel
code review at repository scale
Large
refactors with many touch points

Infinite Subagents Prompt

prompt — infinite subagent pattern
Perform a complete dead code analysis of this repository.

Phase 1: Explore all directories and list every Go source file.
Phase 2: For each source file, generate a task description asking a
  subagent to identify: unexported functions with zero callers,
  unreachable code blocks after return statements, and empty
  error branches. Output format: JSON array {file, line, type, snippet}.
Phase 3: Spawn all subagents from phase 2 in parallel.
Phase 4: Aggregate all results. Deduplicate. Sort by file path.
  Produce a final markdown report with a summary count at the top.
warning Token Budget Warning

The infinite subagents pattern consumes significant API tokens — each subagent uses its own context window, and spawning 50 subagents means 50 separate inference calls. Before launching, estimate the token cost: average tokens per subagent × number of subagents × your model's per-token rate. Set a clear budget ceiling and use this pattern for large codebases or batch operations only — not for tasks where a single focused session would suffice.

Cost Management Strategies

StrategyHow to implement
Scope by directory Limit subagents to a specific subdirectory rather than the full repo
Batch into groups Assign multiple files per subagent rather than one file per subagent
Use a lighter model for subagents Instruct Claude to use a faster/cheaper model variant for leaf-level tasks
Filter before spawning Have the main agent identify which files are most relevant before generating task descriptions
Stage the work Run the pattern on one package at a time, validate quality, then scale
Decision Guide

When to Use Subagents vs a Single Agent

Subagents are not always the right choice. They add coordination overhead and consume extra tokens. Use them when the benefits — parallelism, context isolation, specialization — outweigh those costs.

Decision Criteria

SignalRecommendationReason
Task affects 1–3 closely related files Single agent No benefit from isolation; coordination overhead not worth it
Task spans 10+ independent files Subagents Parallelism reduces wall-clock time significantly
Main context window approaching saturation Subagents Offload work to fresh context windows; prevent quality degradation
Subtasks have no dependencies on each other Subagents True parallelism is safe — no ordering constraints
Subtasks depend heavily on shared state Single agent Subagents cannot see each other; relaying state adds complexity
You need a specialized "expert" for one area Subagent Focused scope + specialized prompt produces higher quality output
Short exploratory task (<5 min) Single agent Subagent launch overhead not justified for quick tasks

Cost-Benefit Analysis

✓ Use Subagents When

examples
Audit 40 Go files for security issues
Generate tests for 20 handler functions
Migrate 15 API endpoints to new format
Review a PR with 30 changed files
Refactor all uses of a deprecated API

✗ Skip Subagents When

examples
Fix one bug in one file
Add a single new function
Refactor one package with tight coupling
Answer a question about the codebase
Implement a small feature (1-3 files)
Tip: A good heuristic: if the task would produce a context window more than 50% full when done sequentially, consider subagents. The context saturation point is where output quality starts degrading — subagents prevent you from ever reaching it.
Mechanics

The Task Tool: How Subagents Are Spawned

Internally, Claude Code spawns subagents by calling the Task tool. Understanding its parameters helps you write prompts that give Claude the right instructions for how to structure subagent work.

Task Tool Parameters

When Claude Code calls the Task tool to spawn a subagent, it passes three key parameters:

ParameterTypePurpose
description string Short human-readable label for this subagent task (shown in UI)
prompt string The full task description the subagent receives — this is its entire context for the job
subagent_type string Agent variant to use (e.g., general-purpose). Controls tool access and isolation level

Worktree Isolation

When a subagent is launched with worktree isolation, it runs in a fresh git worktree — a separate checkout of the repository. This is the safest mode for subagents that write files, because changes in one subagent cannot accidentally overwrite another's work:

promptRequesting worktree-isolated subagents
Spawn subagents in isolated worktrees to implement these features
in parallel without file conflicts:
- Feature A: add user rate limiting to internal/handlers/user.go
- Feature B: add request tracing to internal/middleware/trace.go
- Feature C: update config loading in pkg/config/config.go

Each subagent should work in its own worktree branch and commit
its changes. Return the branch name and a summary of changes made.

Background Execution

Long-running subagents can be dispatched to execute in the background. The main agent continues other work and collects results when subagents finish. This is how true parallelism is achieved:

promptBackground parallel execution
Dispatch all of the following as background subagents simultaneously
and wait for all to complete before reporting:

1. Subagent A: Run go test ./... and capture all failures
2. Subagent B: Run golangci-lint ./... and capture all warnings
3. Subagent C: Check go.mod for outdated dependencies

When all three complete, produce a single report combining findings
sorted by priority: test failures first, lint errors second, dep updates last.
Note: Background subagents are still bounded by the same API rate limits as interactive sessions. Spawning many simultaneous subagents will consume tokens rapidly. Monitor your usage when running large fan-out patterns.
Patterns

Practical Subagent Patterns

Four high-value patterns that demonstrate where subagents provide the most concrete benefit in day-to-day engineering work.

Pattern 1: Research + Implement

Separate research from implementation. A research subagent explores the codebase and produces a findings report; the implementation subagent receives that report as context and builds from it:

promptResearch-then-implement pattern
Step 1 — Research subagent:
  Explore all database interaction code in internal/db/ and internal/store/.
  Map every query, its parameters, and whether it uses parameterized statements.
  Return: JSON {file, function, query, parameterized: bool}

Step 2 — Using the findings from step 1:
  For each non-parameterized query found, spawn an implementation subagent
  to rewrite it to use parameterized statements. Each subagent handles
  one file. Commit each file separately.

Pattern 2: Parallel File Processing

Assign a batch of files to each subagent for parallel processing. Faster than sequential processing and keeps each subagent's context window small:

promptParallel batch processing
List all .go files in internal/handlers/. Group them into 4 equal batches.
Spawn one subagent per batch. Each subagent should:
- Read every file in its batch
- Check for: missing context.Context parameters, unhandled errors,
  empty catch blocks
- Return JSON: [{file, line, issue_type, description}]

After all 4 subagents complete, merge all results, remove duplicates,
and sort by severity.

Pattern 3: Test Generation

One subagent per source file, each responsible for generating comprehensive tests. Parallelism cuts test generation time dramatically on large codebases:

promptParallel test generation
Generate tests for every exported function in internal/auth/.
Spawn one subagent per source file. Each subagent should:
- Read its assigned source file completely
- Identify all exported functions
- Write a corresponding _test.go file with table-driven tests
- Cover: happy path, nil inputs, empty inputs, boundary values
- Use testify/assert for assertions

Return: list of created test files with function coverage count.

Pattern 4: Multi-Stage Pipeline

Chain subagents sequentially — each stage's output becomes the next stage's input. Use for workflows where later stages depend on earlier discoveries:

promptSequential multi-stage pipeline
Execute this 3-stage pipeline:

Stage 1 — Discovery (1 subagent):
  Map all API endpoints in internal/api/. Return a list of:
  {method, path, handler_function, auth_required}

Stage 2 — Analysis (1 subagent per endpoint, parallel):
  For each endpoint from stage 1, spawn a subagent to check:
  - Input validation completeness
  - Auth enforcement
  - Rate limiting applied
  Return: {endpoint, issues: []}

Stage 3 — Report (1 subagent):
  Receive all stage 2 results. Produce an executive summary:
  - Total endpoints audited
  - Critical issues (missing auth, no validation)
  - Medium issues (missing rate limits)
  - Recommended fixes in priority order
Architecture

Context Management with Subagents

The primary reason to use subagents is context protection. As a Claude Code session grows, the context window fills — earlier instructions become less influential, output quality degrades, and responses slow down. Subagents are the antidote: fresh context windows on demand.

How Subagents Protect the Main Context

When the main agent delegates a large workload to a subagent, the work happens in the subagent's context window — not the main one. The main agent only sees the final result, keeping its own context clean:

200K
tokens per context window (Claude 3.x)
~50%
quality degradation threshold — avoid crossing it
N
fresh windows available via subagents

Designing Prompts for Context Efficiency

When writing subagent task descriptions, include only the context the subagent actually needs — not your entire session history. Lean, targeted prompts keep each subagent's context window available for the actual work:

✓ Lean subagent prompt

prompt
File: internal/auth/jwt.go
Task: Check lines 45-120 for weak signing
algorithms and missing expiry validation.
Output: JSON [{line, issue, severity}]

✗ Context-bloated prompt

prompt
[Full session history pasted here]
[All previous findings pasted here]
[Entire codebase structure pasted here]
Now also check internal/auth/jwt.go for
signing issues.

Error Handling and Recovery

Subagents can fail. The main agent receives an error result and must decide whether to retry, skip, or surface the failure. Build recovery logic into your orchestrator prompts:

promptOrchestrator with error handling
Spawn subagents to review each file in internal/handlers/.
If a subagent fails or returns no results for a file, do not abort —
log the failure as {file, error: "subagent failed"} and continue
with the remaining files. At the end, include a "failed_files" section
in the report listing any files that could not be reviewed.
Warning: If a subagent runs out of context mid-task, it will produce a truncated or incomplete result. For very large files or tasks, break the work into smaller chunks — assign line ranges rather than whole files, or split functions rather than entire packages. A completed small result is more useful than a truncated large one.
Real-World Example

Complete Multi-Agent Workflow: Security Audit

A complete end-to-end example showing how to orchestrate a full security audit of a Go web service using multiple specialized subagents. This demonstrates all the patterns — meta prompting, parallel execution, context passing, and result aggregation — in a single workflow.

The Orchestrator Prompt

The user sends this single prompt to the main Claude Code agent. Everything else — decomposition, subagent spawning, synthesis — happens automatically:

promptFull orchestrator prompt
Perform a comprehensive security audit of this Go web service.

Phase 1 — Discovery (single subagent):
  Map the repository structure. List all packages and their purpose.
  Identify the highest-risk areas: auth, input handling, DB access,
  external HTTP calls, file I/O. Return a priority-ranked list.

Phase 2 — Parallel Audit (one subagent per high-risk area from phase 1):
  Each subagent receives its assigned package and this checklist:
  [ ] SQL injection: all queries use parameterized statements
  [ ] Auth bypass: every protected handler checks auth before logic
  [ ] Input validation: all user inputs are validated and bounded
  [ ] Error information leakage: errors do not expose stack traces to clients
  [ ] Hardcoded secrets: no credentials in source files
  [ ] Dependency vulnerabilities: note any obvious outdated packages

  Output per subagent: JSON array
  [{file, line, category, severity, description, recommendation}]

Phase 3 — Synthesis (single subagent):
  Receive all phase 2 results. Produce a final markdown report:

  ## Executive Summary
  - Critical issues: N
  - High issues: N
  - Medium issues: N

  ## Critical Findings
  [Table: File | Line | Issue | Recommendation]

  ## High Findings
  [Table: ...]

  ## Remediation Priority
  Ordered list of fixes by impact × effort score.

What Happens Internally

1
Discovery subagent runs
Reads the repo structure and returns a prioritized list of packages. Takes ~30 seconds.
2
Main agent generates audit prompts
Using the discovery results, the main agent writes one detailed audit task description per high-risk package — this is the meta prompting step.
3
Audit subagents run in parallel
All audit subagents fire simultaneously. Each reads its assigned package and checks the security checklist. Wall-clock time: ~60 seconds regardless of how many packages are audited.
4
Synthesis subagent aggregates
Receives all JSON findings from the audit subagents. Deduplicates, classifies, and produces the final markdown report.
5
Main agent returns the report
The complete security report is displayed to the user. The main agent's context window only held coordination and the final report — not the full audit of every file.
Tip: This same pattern scales from a 5-file microservice to a 500-file monorepo. The only change is the number of audit subagents spawned in phase 2. The orchestrator prompt stays identical — the main agent determines the right number of subagents from what it discovers in phase 1.
