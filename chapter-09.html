Chapter 09 Foundations

What Are Output Styles?

Output styles define how Claude Code communicates — not what it knows, but how it responds. Tone, format, verbosity, and structural preferences are all configurable. The default style is generic. Custom styles make Claude feel like it was built specifically for your workflow.

The Concept: A Style is a Persistent Prompt Addition

An output style is a markdown file that acts as a permanent addition to Claude's system prompt for your session. Every response Claude gives is shaped by the rules in that file — tone, structure, what to include, what to omit. Think of it as a configuration layer that sits between "what Claude knows" and "how Claude talks to you."

Without a style, Claude gives generic, polished, educational responses — fine for beginners, but often too verbose and preamble-heavy for professional engineers who just need answers. With a well-crafted style, Claude behaves exactly like a senior colleague who already knows your preferences.

Where Styles Are Stored

LocationScopeCommitted to repo?
~/.claude/ User-level — applies to all your projects No — personal only
.claude/output-styles/ Project-level — applies only to this repository Yes — shared with team

Built-In Styles

Claude Code ships with three built-in output styles:

  • default — balanced, general-purpose responses
  • explanatory — more context, reasoning visible, educational tone
  • learning — verbose walkthroughs, explains every step, suited for onboarding

Custom styles live as markdown files in .claude/ (project-level) or ~/.claude/ (user-level) and override these defaults per-context.

Note: Style files only affect how Claude responds — they do not change its capabilities, tool access, or model. A "concise" style does not make Claude less capable; it just makes it stop padding responses with unnecessary explanation.
Tone Control
Define whether Claude should be terse or elaborate, formal or casual, declarative or exploratory.
Format Control
Specify preferences for bullet points vs prose, code blocks, table usage, and header structure.
Verbosity Level
Cap explanation length, suppress preamble, eliminate summaries — or expand detail on request.
Project-Level Consistency
Commit a style to the repo so every team member gets the same interaction pattern from Claude Code.
Setup

Creating Custom Output Styles

Use /output-style new to launch a setup subagent that asks questions about your preferences and generates a style file. Or create the markdown file manually for full control.

Using the Command

prompt
/output-style new

The subagent will ask about your preferred response length, formatting conventions, what phrases to avoid, and how you want code references handled. It then writes the style file to .claude/.

Manual Style File

Create a markdown file with your style rules directly. The file name becomes the style identifier. For example, .claude/concise-engineer.md:

markdown — .claude/concise-engineer.md
# Concise Engineer Output Style

You are working with a senior backend engineer who values brevity.

## Rules
- Lead with the answer or action, never with preamble
- Max 3 sentences for explanations unless asked for more
- Use code blocks for all code, even one-liners
- No filler phrases: "Great question!", "Certainly!", "Of course!"
- When referencing code, include file:line_number
- Bullet points over paragraphs for lists
- Skip summaries at the end of responses — the engineer can read the diff

Style File Frontmatter

Style files support optional YAML frontmatter to declare metadata like the style name, description, and whether it inherits from another style:

markdown.claude/output-styles/go-expert.md with frontmatter
---
name: go-expert
description: Go-specific expert style for backend engineering
extends: concise-engineer
---

# Go Expert Output Style

## Additional Rules (extends concise-engineer)
- Always use idiomatic Go patterns
- Prefer table-driven tests with t.Run()
- Reference controller-runtime patterns when relevant
- When suggesting error handling, always wrap errors with context
- For Kubernetes code, follow controller-runtime conventions
- Never suggest global state; prefer dependency injection
Tip: The style file is itself a prompt. Think of it as a persistent system prompt addition that shapes every response Claude gives you. The clearer and more specific your rules, the more reliably Claude adheres to them. Imperative rules ("Always do X", "Never do Y") are more reliable than suggestions ("Try to do X").
Team Workflows

Project-Level Output Styles

Project-scoped styles are stored in .claude/output-styles/ and committed to the repository. Every team member who clones the repo gets the same interaction style without any personal setup.

Setup

1
Create the directory
Create .claude/output-styles/ in your project root.
2
Write the style file
Create a markdown file, e.g. team-concise.md, with your team's communication rules.
3
Commit to the repository
Check the style file into version control. All team members get the style when they pull.
4
Activate the style
Run /output-style project:team-concise to activate it in the current session.

User vs Project Scope

User Style
  • Stored in ~/.claude/
  • Applies across all projects
  • Personal preferences only
  • Not shared with team
  • Activated with /output-style style-name
Project Style
  • Stored in .claude/output-styles/
  • Applies only to this project
  • Team-shared and version-controlled
  • Committed to the repository
  • Activated with /output-style project:name
Interface

Status Line Customization

The status line displays real-time context at the bottom of the Claude Code interface. You can customize it to show model name, token count, cost, session ID, and more — making key metrics visible at a glance without breaking your flow.

Basic Configuration

Set the statusLine field in .claude/settings.json:

json — .claude/settings.json
{
  "statusLine": "${model} | tokens: ${tokens} | cost: $${cost}"
}

Available Variables

VariableDescription
${model}The active model name (e.g., claude-sonnet-4-5)
${tokens}Total tokens used in the current session
${cost}Estimated session cost in USD
${session}Session identifier
${mode}Current mode (auto, manual, etc.)

Advanced: Last User Prompt

For context-at-a-glance workflows, you can display the last user prompt in the status line. This is useful when switching between multiple Claude Code windows and needing to orient quickly without scrolling back through the conversation.

Version Requirement

Status line customization requires Claude Code 1.x+. Check your version with claude --version before configuring advanced status line features.

Future Direction

The Future: Teams of Specialized Agents

Output styles combined with custom skills and subagents form the foundation of agent specialization. Instead of one general-purpose assistant, you compose a team of role-specific agents — each tuned for a distinct part of the development workflow.

Agent Team Architecture

Each agent in the team has a specialized output style matched to its role. The style defines not just format, but the domain knowledge emphasis, what to include or omit, and how to communicate results to the next agent in the pipeline.

Architect
Reads codebase and requirements. Produces structured specs and ADRs. Outputs no implementation code — only design decisions and rationale.
⌨
Implementer
Consumes specs from the Architect. Writes production code. Responds with diffs and implementation notes, not explanations.
Reviewer
Reads code produced by the Implementer. Produces a structured review with severity classifications. No code output — only analysis.
Tester
Generates test cases and coverage reports for the implementation. Outputs test files and a coverage summary with gap analysis.
Where the Industry Is Heading

This is the direction AI-assisted development is moving — not one generalist coding assistant, but a team of role-specialized agents collaborating on your codebase. Output styles are what give each agent its distinct identity and communication contract.

Examples

Practical Style Examples

Four complete, production-ready style files — each tuned to a different engineering role. Copy, adapt, and commit these to your project or user configuration.

Style 1: Concise — Minimal, Direct Answers

The workhorse style for experienced engineers who want answers without preamble. Eliminates all padding, filler phrases, and post-amble summaries:

markdown~/.claude/concise.md
---
name: concise
description: Minimal, direct responses for experienced engineers
---

# Concise Style

## Communication Rules
- Answer first, explain only if the answer requires it
- Maximum 3 sentences of explanation per response unless explicitly asked
- No preamble: never start with "Great question", "Certainly", "Sure!", "Of course"
- No postamble: no "Let me know if you need anything else", no summary recaps
- No filler words or hedging: be direct

## Code Rules
- Use code blocks for all code including single-line commands
- Include file:line references when pointing to specific code locations
- Show diffs rather than full file rewrites when only part of a file changes

## Formatting
- Prefer bullet lists over paragraphs for multi-point answers
- Use tables for comparisons
- Omit headers unless the response genuinely has multiple sections

Style 2: Verbose — Full Context and Reasoning

For onboarding, learning, or when you need Claude to explain its reasoning fully. Every step is explained, tradeoffs are surfaced, and alternatives are noted:

markdown~/.claude/verbose.md
---
name: verbose
description: Educational, fully-explained responses with visible reasoning
---

# Verbose Style

## Communication Rules
- Always explain WHY before HOW — rationale before implementation
- Surface alternatives and their tradeoffs before recommending one approach
- Explicitly state assumptions you are making
- Call out risks and edge cases even if not asked

## Code Rules
- Annotate code with comments explaining non-obvious decisions
- After code, include a "What this does" paragraph in plain English
- Include both the implementation and a usage example
- Mention what to test after any code change

## Formatting
- Use headers to separate major sections of long responses
- Include a TL;DR at the top of responses longer than 10 lines
- Use numbered steps for sequential processes
- Bold key terms on first use

Style 3: Go Expert — Language-Specific Engineering

Specializes Claude for idiomatic Go development. Enforces Go conventions, controller-runtime patterns, and project-specific standards:

markdown.claude/output-styles/go-expert.md
---
name: go-expert
description: Idiomatic Go engineering style for backend and Kubernetes work
extends: concise
---

# Go Expert Style

## Go Conventions
- Always handle errors explicitly — never use blank identifier _ for errors
- Wrap errors with context: fmt.Errorf("operation failed: %w", err)
- Use context.Context as first parameter in all I/O functions
- Prefer interfaces for dependency injection and testability
- Use table-driven tests with t.Run() subtests
- Avoid global state — pass dependencies explicitly

## Kubernetes / controller-runtime
- Use ctrl.Result{} correctly — do not requeue unnecessarily
- Use client.Client for API server interactions, never raw clientset
- Handle IsNotFound errors with apierrors.IsNotFound(err)
- Use controllerutil.SetControllerReference for owned resources
- Distinguish between .Owns() (direct) and .Watches() + EnqueueRequestsFromMapFunc (indirect)

## Code Output
- Show idiomatic Go, never Java-style patterns
- Imports: standard library first, then third-party, then internal
- File references: always include package name and line number
- When suggesting tests: show Ginkgo/Gomega patterns if existing tests use them

Style 4: Security Reviewer — Audit-Focused

Tunes Claude to behave as a security-focused code reviewer. Every response is framed around risk identification, severity classification, and remediation:

markdown.claude/output-styles/security-reviewer.md
---
name: security-reviewer
description: Security-focused review style for audit and threat modeling
---

# Security Reviewer Style

## Mindset
- Default to adversarial thinking: assume inputs are malicious
- Classify every finding: CRITICAL / HIGH / MEDIUM / LOW / INFO
- For each issue, state: what it is, how it can be exploited, how to fix it
- Surface issues even if they are edge cases — better to over-report

## Review Checklist (apply to all code reviews)
- Input validation: are all user inputs validated and bounded?
- SQL injection: are all queries parameterized?
- Auth bypass: is every protected route checking auth before logic?
- Information leakage: do error messages expose internals?
- Hardcoded credentials: any secrets, tokens, or keys in source?
- Dependency risk: any outdated packages with known CVEs?

## Output Format
- Findings as a table: Severity | File:Line | Issue | Recommendation
- Summary line: "X critical, Y high, Z medium findings"
- If no issues found: "No issues found in scope: [list what was checked]"
- Never say "looks good" without explicitly stating what was checked

✓ Effective Style Rules

  • Never start with preamble
  • Always include file:line references
  • Use table format for findings:
    Severity | File | Issue | Fix
  • Classify severity as CRITICAL/HIGH/MEDIUM/LOW
  • Maximum 3 sentences per explanation

✗ Ineffective Style Rules

  • Be concise
  • Be helpful
  • Focus on security
  • Try to be direct
  • Avoid being too verbose
    unless needed
Note: Vague directives like "be concise" are less reliable than specific rules like "maximum 3 sentences". Claude interprets "concise" differently in different contexts. Specific, measurable rules produce consistent behavior across all responses.
Decision Guide

When to Use Which Style

Choosing the right style for the context saves time and produces better output. Use this as a decision reference.

SituationRecommended StyleReason
Daily feature development by senior engineers concise Minimizes noise; assumes you can read code
Onboarding a new team member verbose / learning Full explanations help them learn the codebase
Go/Kubernetes projects go-expert Enforces idiomatic patterns specific to the language
Pre-release security audit security-reviewer Adversarial lens; structured findings; nothing missed
Writing ADRs or documentation verbose Detailed reasoning makes better docs
Debugging a tricky bug verbose You want to understand WHY, not just the fix
Automated CI pipeline concise (or custom machine-readable) Output needs to be parseable; no prose needed

Combining Styles with CLAUDE.md

Output styles and CLAUDE.md serve different purposes and work together. CLAUDE.md provides project context — what the project does, how to build it, which patterns to follow. An output style shapes how Claude communicates that knowledge back to you. Both can be active simultaneously:

markdownCLAUDE.md — provides project context
# CLAUDE.md

## Project
KubeVela controller — Go 1.23, controller-runtime v0.17

## Build Commands
make generate && make manifests

## Architecture
Uses ResourceTracker pattern instead of .Owns() for child resources.
PostDispatch traits depend on workload runtime status.
markdown.claude/output-styles/go-expert.md — shapes how Claude talks
---
name: go-expert
---
- Always use idiomatic Go
- Reference controller-runtime patterns
- Include file:line on all code references

With both active: Claude knows the project architecture from CLAUDE.md and communicates in the style defined by go-expert.md. The combination is more powerful than either alone.

Tip: Start with a simple concise style for personal use and a project-specific style committed to the repo for team use. Add specialization gradually — it is easier to add rules than to remove ones that are causing unexpected behavior. Test your style by asking Claude a routine question and evaluating whether the response matches your intent.
