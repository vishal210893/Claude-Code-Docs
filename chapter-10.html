Chapter 10 Foundations

What Are Agent Skills?

Skills are reusable, shareable custom slash commands that encapsulate complex workflows. Stored as markdown files, they are invoked by name and their content is inserted as a user message — giving Claude detailed, consistent instructions every time.

How Skills Work

A skill is a prompt template. When you run /skill-name, Claude Code reads the corresponding markdown file and inserts its content into the conversation as your message. The skill file is the instruction set — you are the one who encoded your best practices into it once, so you never have to repeat yourself.

Skills are stored as markdown files:

  • .claude/commands/ — project-scoped, committed to the repo
  • ~/.claude/commands/ — user-scoped, available in every project
Note: Skills are also called "custom slash commands" in the Claude Code documentation. The term "skill" emphasizes that they encode expertise — not just shortcuts. Every skill you write is a piece of team knowledge made executable.

Skills vs Slash Commands vs MCP — The Quick Distinction

These three concepts often get confused because they all extend what Claude Code can do:

  • Skills (custom slash commands) — markdown files you write that become /commands. Zero infrastructure. No runtime. Just text instructions.
  • Built-in slash commands — commands like /help, /clear, /todo that are built into Claude Code itself. You cannot redefine these.
  • MCP (Model Context Protocol) — a separate server process that exposes structured tools and live data. Requires a running process, network communication, and a schema definition.

The decision rule is simple: if your workflow can be expressed as a sequence of instructions in a markdown file, use a skill. If you need to query a live API, database, or external system and get a structured response back, use MCP.

Tip: Start with a skill. You can always upgrade to MCP later if you outgrow it. Most teams discover that skills cover 80% of their automation needs with 5% of the setup effort of MCP.

When Claude Auto-Invokes vs Waits

Skills are always explicitly user-invoked via the slash command — Claude Code does not auto-invoke skills on its own. However, you can reference skills in your CLAUDE.md to suggest when they should be used. For example:

markdownCLAUDE.md — skill usage hint
## Workflow Guidelines
When a task is complete and ready for merge, suggest running /create-pr.
Before any deployment, run /pre-deploy-check.

With this in CLAUDE.md, Claude will suggest the relevant skill at the right moment — but you still type the command to trigger it.

Setup

Creating Skills: Step-by-Step

Create a .md file in .claude/commands/. The file name becomes the slash command. The file content becomes the prompt that Claude Code executes when you invoke it.

Directory Setup

bashCreate the commands directory
# Project-scoped skills (committed to repo, shared with team)
mkdir -p .claude/commands

# User-scoped skills (personal, available in every project)
mkdir -p ~/.claude/commands
Note: The file name becomes the slash command. .claude/commands/create-pr.md becomes /create-pr. Use hyphens for multi-word names — they become hyphens in the slash command. Spaces are not supported in command names.

Skill File Structure

A skill file is a standard markdown file. There is no special frontmatter required — the entire file content becomes the prompt. However, a good skill file follows a consistent structure:

markdownAnatomy of a skill file
# Skill Name (H1 — becomes the skill title in /help)

One paragraph describing what this skill does and when to use it.

## Steps (optional — for sequential workflows)
1. First step with specific tool instructions
2. Second step building on the first
3. ...

## Acceptance Criteria (optional — tells Claude when it's done)
- The output must include...
- The result must pass...

## Format (optional — tells Claude how to format output)
Return results as a markdown table with columns: ...

Example: PR Creation Skill

Create .claude/commands/create-pr.md to invoke with /create-pr:

markdown.claude/commands/create-pr.md
# Create Pull Request

Analyze the git diff between current branch and main.

Steps:
1. Run `git log --oneline main..HEAD` to see commits
2. Run `git diff main...HEAD --stat` for changed files
3. Create a PR with:
   - Title: [type]: description (max 72 chars)
   - Summary: 3 bullet points max
   - Problem: why this change was needed
   - Testing: how to verify it works
   - Breaking changes: if any

Example: Deploy Skill with Pre-Flight Checks

markdown.claude/commands/deploy.md
# Deploy to Staging

Target environment: $ARGUMENTS (default: staging)

Pre-flight checklist — verify each before proceeding:
1. Run `go test ./...` — all tests must pass
2. Run `go build ./...` — build must succeed
3. Run `git status` — no uncommitted changes
4. Check `git log --oneline origin/main..HEAD` — confirm commits to deploy

If all checks pass, proceed:
5. Run `make docker-build` to build the image
6. Run `make deploy ENV=$ARGUMENTS` to deploy
7. Run `kubectl rollout status deployment/app -n staging` to verify rollout
8. Report the deployed version and pod status

Example: Code Review Skill

markdown.claude/commands/review.md
# Code Review

Review the code changes specified by $ARGUMENTS (or the current branch diff if no argument).

Run: `git diff main...HEAD -- $ARGUMENTS`

Review for:
1. **Correctness** — does the logic achieve what the commit messages claim?
2. **Edge cases** — are nil/empty/boundary conditions handled?
3. **Error handling** — are all errors checked and handled appropriately?
4. **Security** — input validation, no hardcoded secrets, no SQL injection risk
5. **Tests** — are new functions covered by tests?
6. **Naming** — are identifiers clear and consistent with the codebase style?

Output format:
- Critical issues (must fix before merge)
- Suggestions (optional improvements)
- Positive observations (what was done well)
Tip: Skills support a $ARGUMENTS placeholder. Use /skill-name some args and $ARGUMENTS in your skill file will be replaced with some args. This lets a single skill file handle parameterized invocations — for example, /review-pr 142 where 142 is passed as the PR number.

Best Practices for Writing Effective Skill Descriptions

  • Be specific about outputs — tell Claude exactly what format to return results in. Vague instructions produce inconsistent outputs.
  • Encode acceptance criteria — skills are most powerful when they define "done." Include explicit success conditions so Claude knows when to stop.
  • Reference concrete commands — write the exact shell commands you want Claude to run, rather than describing them abstractly. Run: `go test ./...` is better than "run the tests."
  • Order steps carefully — Claude follows the order of steps in your skill file. Put validation and read-only checks before write operations.
  • Keep skills focused — one skill, one workflow. A skill that tries to do three different things is harder to maintain and produces unpredictable results.
Warning: Skills do not restrict what tools Claude Code can use. A skill that says "review the code" can still trigger file writes if the model decides that is appropriate. If you need to restrict Claude's actions, use permission settings in .claude/settings.json, not the skill file itself.
Advanced

Custom Skills with Auxiliary Scripts

Skills can reference and run scripts. The pattern is powerful: the skill file describes the workflow and provides Claude with the analytical task, while a script handles the deterministic, structured work — scanning, linting, coverage checks — that Claude interprets.

Script-Backed Skill Pattern

Store scripts in .claude/scripts/ and reference them from your skill file:

markdown.claude/commands/security-audit.md
# Security Audit

First, run: `bash .claude/scripts/security-scan.sh`

Then analyze the output and:
1. Categorize findings by severity (Critical/High/Medium/Low)
2. For each Critical/High finding, suggest a specific fix
3. Output a structured report with actionable remediation steps

Example: Coverage Report Skill

A script gathers coverage data; the skill instructs Claude what to do with it:

bash.claude/scripts/coverage.sh
#!/bin/bash
go test -coverprofile=coverage.out ./...
go tool cover -func=coverage.out | tail -1
go tool cover -func=coverage.out | grep -E "^[^t].*[0-9]\.0%$" | head -20
markdown.claude/commands/coverage.md
# Test Coverage Analysis

Run: `bash .claude/scripts/coverage.sh`

Analyze the output:
1. Report overall coverage percentage
2. List the 10 packages with the lowest coverage
3. For each low-coverage package, identify which functions lack tests
4. Suggest which 3 functions would provide the highest value to test next,
   based on complexity and criticality

Skill Composition: Skills That Reference Other Skills

You can document in a skill file that the user should run another skill first — creating a manual composition chain. This is not automatic, but it encodes the correct workflow order for your team:

markdown.claude/commands/release.md
# Release Workflow

This skill runs the full release sequence. Before starting, ensure
`/pre-deploy-check` has passed cleanly.

Steps:
1. Run `git tag -a v$ARGUMENTS -m "Release v$ARGUMENTS"`
2. Run `git push origin v$ARGUMENTS`
3. Run `make release-notes VERSION=$ARGUMENTS`
4. Create the GitHub release with the generated notes
5. Monitor the CI pipeline until the release artifacts are published

Acceptance criteria:
- Git tag exists on origin
- GitHub release is published
- CI pipeline shows green for the release workflow
Note: Scripts in .claude/scripts/ are a great place to put deterministic checks — linting, dependency scanning, test coverage reports — that feed into Claude's analysis. The script does the data gathering; Claude does the interpretation and recommendation.
Tip: Make your scripts exit with non-zero codes on failure. Claude Code will see the failed exit code in the Bash tool result and will naturally report the failure rather than silently treating bad output as success.
Examples

Real-World Skill Examples

The following are complete, production-ready skill files. Each encodes a real workflow that teams use repeatedly. Copy and adapt them to your project's conventions.

Skill 1: Incident Debug

Used when an alert fires and you need a fast investigation. Pass the service name or error message as $ARGUMENTS.

markdown.claude/commands/debug-incident.md
# Debug Incident

Investigate the issue: $ARGUMENTS

Investigation steps:
1. Search for recent changes related to this area:
   `git log --oneline --since="3 days ago" -- $ARGUMENTS`
2. Check current error patterns in logs (if log file exists):
   `tail -100 logs/app.log | grep -i "error\|panic\|fatal"`
3. Read the relevant source files for the component mentioned
4. Check if any config values look suspicious
5. Look for recent dependency changes in go.sum or package-lock.json

Output:
- Root cause hypothesis (most likely cause based on evidence)
- Evidence supporting the hypothesis
- Immediate mitigation steps (what to do right now)
- Fix recommendation (what code change resolves this permanently)

Skill 2: Database Migration Scaffolding

markdown.claude/commands/new-migration.md
# New Database Migration

Create a database migration for: $ARGUMENTS

Steps:
1. Check the latest migration file in `db/migrations/` to determine the
   next version number
2. Create a new migration file: `db/migrations/NNNN_$ARGUMENTS.sql`
3. Write the UP migration (schema change)
4. Write the DOWN migration (rollback) — this is required, not optional
5. Create a corresponding struct update in the appropriate model file
   if this migration adds or removes columns from an existing table
6. Add a test that verifies the migration applies cleanly

Constraints:
- Every migration must be reversible
- Column names must use snake_case
- Foreign key constraints must be named explicitly

Comparison

Skills vs MCP

Skills and MCP solve different problems. Skills encode workflows and best practices into reusable prompts. MCP provides structured access to live external systems. Choosing the right tool depends on whether you need static workflow instructions or dynamic external data.

Comparison Table

Aspect Skills MCP
What it is Prompt template / slash command External tool server with structured APIs
Setup complexity Low — just a .md file Medium-High — separate server process
Capabilities Text instructions + file access Real-time data, external APIs, databases
Persistence Stateless — each invocation fresh Can maintain state in server
Best for Repeatable workflows, best practices External integrations, live data
Distribution Commit to repo Publish npm package or run server
Decision Rule

Choose Skills when you are encoding a workflow. Choose MCP when you need real-time external data or structured tool APIs that return dynamic results.

Comparison

Skills vs Subagents

Skills are user-initiated prompt templates. Subagents are programmatically spawned by the main agent to run work in parallel isolated context windows. The distinction is who controls invocation and whether parallelism is needed.

Comparison Table

Aspect Skills Subagents
Invocation Slash command by user Spawned by main agent programmatically
Context Shares main agent context Isolated context window
Parallelism Sequential Parallel — multiple subagents at once
Control User-initiated Agent-initiated
Best for Repeatable user workflows Parallel agent tasks, batch operations
Which to Use

Use Skills for things you want to invoke manually on demand. Use Subagents when the main agent needs to delegate and parallelize work without user involvement.

Guidance

Skills Tradeoffs

Skills are intentionally simple. That simplicity is a strength in most cases and a limitation in others. Understanding where skills excel and where they fall short prevents choosing the wrong abstraction.

When to Use Skills
  • Code review templates
  • PR creation and description generation
  • Deployment checklists
  • Documentation generation
  • Repeatable analysis workflows
  • Team-standard task sequences
  • Anything version-control distributable
When Not to Use Skills
  • Querying live databases
  • Calling external REST APIs
  • Maintaining conversation state across runs
  • Real-time data retrieval
  • Tasks needing structured tool responses
  • Anything requiring server-side logic

Good vs Anti-Pattern

✓ Good Practice

markdownFocused, actionable skill
# Create PR
1. Run `git log --oneline main..HEAD`
2. Run `git diff main...HEAD --stat`
3. Create PR with title format: type: description
4. Add summary, problem, and testing sections

✗ Anti-Pattern

markdownVague, over-broad skill
# Help me
Do whatever needs to be done to
get this code ready. Review it,
fix bugs, write tests, make a PR,
and deploy if everything looks good.

Summary

  • Pros: dead simple to create, no infrastructure required, shareable via git, fully version-controlled, zero runtime dependencies
  • Cons: text-only instructions, no structured API responses, no real-time data access, no state persistence between invocations

Use concrete shell commands, not abstract descriptions Claude will run the exact command rather than guessing what command to use
Define an explicit output format Produces consistent, parseable results across invocations
Include acceptance criteria ("done when...") Prevents Claude from continuing indefinitely or stopping too early
One skill per workflow — keep them focused Easier to maintain, test, and reason about
Commit project skills to the repo Team gets them automatically; they evolve with the codebase
Test new skills with a dry run Run the skill on a branch first — verify it behaves as expected before using on main
