Chapter 11 Overview

Claude Code Desktop

Claude Code Desktop is the claude.ai desktop application with deep Claude Code integration. It enables background agents that continue working while you focus elsewhere, and provides a unified interface for managing multiple parallel agent sessions across local and cloud execution environments.

Two Execution Modes

Desktop exposes two fundamentally different ways to run Claude Code agents:

  • Local Mode — the agent runs on your machine, with full access to your filesystem, environment variables, and local tools
  • Cloud Mode — the agent runs on Anthropic's infrastructure, always-on, no local machine dependency

You can run both simultaneously — a local agent working on filesystem-heavy tasks alongside a cloud agent handling long-running background work.

Note: Claude Code Desktop is distinct from the Claude Code CLI. The CLI (claude in your terminal) is the command-line interface. The Desktop app provides a GUI wrapper around similar functionality, adding background agents, session management, and cloud execution. Both use the same underlying model and settings.

Configuration

Local vs Cloud Agents

Understanding the capabilities and constraints of each mode lets you route tasks to the right execution environment. Most development workflows benefit from using both modes together.

Local Mode
  • Runs on your machine
  • Full filesystem and environment access
  • Uses your API key and local tools
  • Zero latency for file operations
  • Requires your machine to be running
  • Best for file-heavy, short-to-medium tasks
Cloud Mode
  • Runs on Anthropic's infrastructure
  • Sandboxed — no local filesystem access
  • Per-token + compute cost
  • Always-on — machine can be off
  • Best for long-running background tasks
  • Great for 24/7 automation workflows

Core Feature

Background Agents

Background agents are Claude Code instances running detached from your active session. You give them a task with clear acceptance criteria, send them to the background, and continue working. They notify you on completion or when a decision is needed.

Workflow

1
Open Claude Code Desktop
Launch the desktop app and start a new session with a clearly scoped task description and explicit acceptance criteria.
2
Define the task
Write a complete task description. The more specific the acceptance criteria, the less the agent will interrupt you for clarification.
3
Send to background
Use ⌘+Shift+B or the session menu to detach the agent. It continues running while you work on something else.
4
Continue with other work
Open a new session, work in your editor, or close the window. The agent is not affected.
5
Review results
The agent notifies on completion or when it reaches a decision point that requires your input.

Writing Effective Background Task Descriptions

Background agents work best with upfront clarity. Unlike interactive sessions where you can steer mid-task, a background agent will make its own choices when it hits ambiguity. A good background task description includes:

textBackground task template
Goal: [One sentence describing the end state]

Scope: [Exactly what files/packages/areas to touch]

Constraints:
- Do NOT modify [specific files or areas]
- Do NOT add new dependencies without noting them in CHANGES.md
- All tests must pass after each file change

Acceptance criteria:
- [Specific, verifiable outcome 1]
- [Specific, verifiable outcome 2]

If blocked on a decision, write your question to QUESTIONS.md and stop.
Tip: The "write questions to a file and stop" instruction is the single most important addition to background task prompts. Without it, an agent that hits an ambiguous decision will either make a wrong choice silently or ask you at an inconvenient moment. Redirecting its confusion to a file gives you async communication.

Monitoring Long-Running Agents

You do not need to watch an agent in real time, but periodic check-ins improve outcomes on long tasks:

  • Check the todo list — open the session and run /todo to see what the agent has done and what remains
  • Review git status — the agent's changes are visible via git diff even while it is running
  • Watch for QUESTIONS.md — if you asked the agent to write questions there, check the file periodically
  • Set time-based check-ins — for tasks longer than 30 minutes, check in every 15-20 minutes

When to Intervene

Stop a background agent and redirect it when you observe:

  • It is making changes outside its stated scope
  • It has been running for significantly longer than expected
  • The todo list shows it created many new items not in the original plan
  • Git diff shows changes to files you did not expect it to touch

High-Value Use Cases

  • Large refactors — rename a type across the codebase, migrate an API pattern
  • Test generation — write coverage for an entire package while you work on the next feature
  • Documentation — generate or update docs across a repo
  • Research tasks — read and summarize a large codebase or set of files
  • Dependency upgrades — update a major dependency version across all affected files
  • Code style migration — apply a new linting rule across the entire codebase
Warning: Background agents can accumulate significant token costs on long-running tasks. Set a budget cap before starting, and check in periodically on progress. Overly broad tasks with no scope limit are the most common cause of unexpected costs.
Parallel Development

Git Worktrees

Git worktrees give you multiple working directories from the same repository, each with a different branch checked out simultaneously. This is the foundation for running parallel Claude Code sessions without context interference between tasks.

Worktrees vs Branches

A branch switch changes what your single working directory contains. A worktree gives you a separate directory with a different branch checked out — both coexist at the same time. You can open Claude Code in each directory independently, with no risk of one session affecting the other's files.

Note: Worktrees share the same git object store — they are not clones. Commits made in a worktree are immediately visible to other worktrees via the shared .git directory. Only the working directory and index are separate. This means merges, rebases, and log commands work correctly across all worktrees.
1
Create the worktree
Run git worktree add ../myrepo-feature feature/auth-redesign. A new directory is created with that branch checked out.
2
Open Claude Code in the new directory
Launch Claude Code from the worktree directory. It operates in full isolation from your main working tree.
3
Work in isolation
Changes in the worktree have no effect on your main directory. Both can run active Claude Code sessions simultaneously.
4
Merge and clean up
Commit all changes in the worktree, merge the branch, then run git worktree remove ../myrepo-feature.

Common Commands

bashGit worktree commands
# Create a worktree for a new branch
git worktree add ../project-feat-auth -b feature/auth-redesign

# List all worktrees
git worktree list

# Remove a worktree (after merging)
git worktree remove ../project-feat-auth

# Prune stale worktree records
git worktree prune

Worktrees for Parallel Claude Code Sessions

The key workflow for using Desktop with worktrees to run tasks in parallel:

bashSet up three parallel worktrees
# Create three worktrees for three independent features
git worktree add ../myapp-feat-auth    -b feature/auth
git worktree add ../myapp-feat-api     -b feature/api-v2
git worktree add ../myapp-fix-metrics  -b fix/metrics

# Open Claude Code Desktop in each — all run independently

Each Claude Code session in Desktop gets its own context window, its own branch, and its own working directory. Changes in one cannot affect the others — the isolation is physical, not just logical.

Tip: Name worktree directories with a prefix matching the repo name for clarity: myapp-feat-auth, myapp-fix-bug-123. This makes it obvious which repo and which task each directory belongs to when you have multiple open simultaneously.
Warning: You cannot have the same branch checked out in two worktrees simultaneously. If you try, git will refuse. Each worktree must be on a unique branch. This is by design — git cannot track two different sets of changes to the same branch at the same time.
Workflow

Merging Worktrees

Completing work in a worktree follows a predictable sequence: commit in the worktree, merge back in the main tree, remove the worktree. Resolve conflicts inside the worktree before merging — the isolated environment makes it clear exactly what changed.

Merge Sequence

bash
# Step 1: In the worktree — commit all changes
git add -p
git commit -m "feat: implement auth redesign"

# Step 2: Back in main working tree — merge the branch
git merge feature/auth-redesign

# Step 3: Remove the worktree
git worktree remove ../project-feat-auth

# Step 4: Delete the branch (optional)
git branch -d feature/auth-redesign
Resolve Conflicts Early

Resolve conflicts in the worktree before merging, not after. The isolated environment makes it easier to identify exactly what changed and why — you have full context from the Claude Code session that produced the changes.

Advanced

Orchestrating Parallel Local & Cloud Agents

For large tasks with independent components, run multiple agents simultaneously — each working in its own scope. A local agent handles the filesystem-heavy work, a cloud agent handles the long-running background task, and a main agent synthesizes the results.

Example Architecture

A large feature might run three simultaneous agents:

  • Frontend agent (local, worktree) — implements UI components
  • Backend agent (local, worktree) — implements API endpoints
  • Docs agent (cloud, background) — generates and updates documentation

Each works independently. The main agent collects their outputs and synthesizes the final integration — reviewing cross-cutting concerns and resolving any interface mismatches.

Git as the Integration Bus

Git is what makes parallel agents composable. Each agent works in its own branch or worktree. Merging is the integration step. Conflicts at merge time surface exactly where agent outputs diverge — which is exactly where a human review adds the most value.

Advanced

Remote Development Patterns

Claude Code integrates naturally with remote development environments — SSH remotes, dev containers, and cloud development machines. The key principle is that Claude Code runs where your code lives, not necessarily where your IDE runs.

SSH Remote Workflow

When your development environment is on a remote server — a cloud VM, a shared development server, or a Kubernetes pod — run Claude Code directly on the remote:

bashSSH into remote and run Claude Code there
# SSH to remote dev machine
ssh dev-server.example.com

# Navigate to the repo (already checked out on remote)
cd /home/user/myapp

# Run Claude Code directly on the remote
claude "Add rate limiting to the API handlers"

This pattern gives Claude Code direct access to the remote filesystem, local environment variables, and any tools installed on the remote machine — without any file syncing or network overhead.

Dev Container Workflow

Dev containers standardize the development environment. Claude Code works inside a dev container just as it does on your local machine:

bashClaude Code inside a dev container
# Option A: Open VS Code in devcontainer, use integrated terminal
# The terminal runs inside the container — claude has full container access

# Option B: exec into a running container directly
docker exec -it myapp-dev bash
claude "Run the tests and fix any failures"
Tip: When running Claude Code inside a dev container, the CLAUDE.md file in the repo is automatically available. Add container-specific commands (e.g., how to run the database, how to run migrations) to CLAUDE.md so Claude Code knows the container's development workflow without being told each time.

Cost and Performance Considerations

Remote execution means every file read and shell command result travels over the network between the remote machine and the Anthropic API. For large codebases on high-latency connections, this can noticeably affect task duration. Mitigations:

  • Use .claudeignore to exclude large directories (build artifacts, node_modules, vendor/) from file reads
  • Run Claude Code on a machine geographically close to your code — not from a laptop on the other side of the world
  • For very large repos, pre-summarize key files in CLAUDE.md so Claude spends fewer tokens on initial exploration
Mobile

Claude Code Mobile

Claude Code is available on iOS and Android for review and oversight of agent work. Mobile is optimized for the "check in and approve" workflow — not for driving complex coding sessions, but for staying connected to what your agents are doing.

Mobile Use Cases

  • Reviewing agent progress — check what a background cloud agent has done while you were away
  • Approving plans — an agent paused for confirmation can be unblocked from your phone
  • Reading diffs — review code changes before they get merged
  • Simple interactions — quick questions, context additions, clarifications

Mobile vs Desktop Comparison

Capability Mobile Desktop
Check agent progress Yes Yes
Approve pending decisions Yes Yes

Note: Mobile is best for "check in and approve" workflows where background cloud agents handle the heavy lifting. Complex coding sessions, multi-file edits, and context-heavy debugging belong on desktop where you have full keyboard and screen real estate.
