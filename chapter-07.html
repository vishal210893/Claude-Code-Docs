Chapter 07 Safety

Plan Mode

Plan Mode is a read-only research and analysis mode that prevents Claude Code from making any changes until you explicitly approve a spec. It mirrors how senior engineers work: gather requirements, design the solution, then implement — never the other way around.

Activating Plan Mode

You can enter Plan Mode in two ways:

  • Keyboard shortcut: Shift+Tab+Tab while in the Claude Code prompt
  • Slash command: type /plan at the start of your message

While in Plan Mode, Claude Code will read files, explore the codebase, ask clarifying questions, and produce a structured specification — but it will not create, modify, or delete any files until you exit Plan Mode and approve the plan.

Read-Only Safety
No files are created, modified, or deleted while in Plan Mode. Claude Code can only read and analyze.
Structured Specs
Output is a detailed implementation plan — files to change, functions to add, risks identified — not loose prose.
Senior Workflow
Mirrors how experienced engineers approach problems: design before coding. Prevents over-eager implementation of the wrong solution.
Approval Gate
You review and approve the plan before a single line of code is touched. Disagree with part of the plan? Correct it first.

Plan Mode in Practice

Here is an example of entering Plan Mode with a complex task and the kind of structured output it produces:

prompt — entering plan mode
/plan

Add rate limiting to the public API. Requirements:
- Per-user rate limits (100 req/min for free tier, 1000 req/min for paid)
- Return 429 with Retry-After header when limit is exceeded
- Store rate limit state in Redis with sliding window algorithm
- Expose current limit/remaining/reset in response headers on every request
- Admin endpoint to query and override limits per user
plan output — example
## Plan: API Rate Limiting

### Files to create
- internal/ratelimit/limiter.go     # sliding window algorithm, Redis backend
- internal/ratelimit/middleware.go  # HTTP middleware, header injection
- internal/ratelimit/limiter_test.go

### Files to modify
- internal/server/router.go         # register middleware on public routes
- internal/admin/handlers.go        # add /admin/ratelimit GET + DELETE
- pkg/config/config.go              # add RateLimitConfig struct

### Key decisions
1. Sliding window via Lua script in Redis (atomic, no race conditions)
2. User tier read from JWT claims — no DB lookup per request
3. X-RateLimit-Limit / X-RateLimit-Remaining / X-RateLimit-Reset headers

### Risks identified
- Redis unavailability: fail-open strategy (allow requests, log warning)
- Clock skew: use Redis server time, not application server time

### Approve this plan to begin implementation.
Plan Mode vs Normal Mode

In normal mode, Claude Code will start implementing as soon as it has enough information. Plan Mode forces a design checkpoint. Use it whenever the task touches multiple files, involves irreversible changes (schema migrations, API contract changes), or when you want a second opinion on the approach before any code is written.

Reasoning

Extended Thinking

Extended thinking causes Claude to reason internally before producing a response — consuming more compute and tokens, but significantly improving output quality for complex problems. Trigger it with specific keywords embedded in your prompt.

Thinking Trigger Keywords

Claude Code recognizes four escalating thinking triggers. Use them by including the phrase naturally in your prompt:

Trigger keyword Thinking intensity Best for
think Standard Routine decisions, minor design choices, straightforward debugging
think hard Extended Architecture choices, API design, performance tradeoffs
think harder Deep Complex algorithms, concurrency issues, subtle correctness problems
ultrathink Maximum Critical system design, security architecture, irreversible infrastructure decisions

Using Thinking Keywords

Include the keyword in your prompt naturally — Claude Code recognizes it as a signal to allocate more reasoning budget:

prompt — standard think
Think about the best approach to add pagination to the user list endpoint,
then implement it. The table has ~50k rows.
prompt — ultrathink
Ultrathink about the distributed locking strategy for our order processing
system. We have 3 replicas, orders must be processed exactly once, and
we cannot afford duplicate charges. Design the full locking approach before
writing any code.
Token Cost vs Quality Tradeoff

Extended thinking consumes significantly more tokens than standard responses — ultrathink can use 10–20x more tokens than a default reply. Reserve the higher tiers for genuinely hard problems: architecture decisions you will live with for years, security designs, or algorithms where correctness is critical. Use think for routine decisions to keep costs reasonable.

When to Use Each Level

Use think / think hard for
  • Choosing between two implementation approaches
  • Diagnosing a non-obvious bug
  • Deciding on an API shape
  • Refactoring a single module
Use think harder / ultrathink for
  • Distributed system consistency guarantees
  • Security threat modeling
  • Database schema with complex constraints
  • Performance-critical algorithm design
Version Control

Git-First Development

Claude Code understands git context automatically — current branch, staged changes, recent commits, and remotes. A git-first discipline means every task starts on a branch, progresses through atomic commits, and ends with a PR, giving you full rollback capability at every step.

Branch-Based Workflow

Never work directly on main. Claude Code reads your current branch and uses it to scope its understanding of what work is in progress:

bash — git workflow Claude Code executes
# Start a feature — Claude Code creates and switches to the branch
git checkout -b feature/add-rate-limiting

# Claude implements in stages, committing checkpoints
git add internal/ratelimit/
git commit -m "feat(ratelimit): add sliding window algorithm with Redis backend"

git add internal/server/router.go
git commit -m "feat(ratelimit): wire middleware into public API routes"

# Open PR when done
gh pr create --title "feat: add per-user rate limiting with Redis" --base main

Commit Checkpoint Prompting

Instruct Claude Code to commit at logical checkpoints rather than committing everything at the end:

prompt
Implement rate limiting as planned. After each logical unit is complete
and tested, commit with a conventional commit message before moving to
the next unit. Do not create one giant commit at the end.
Atomic Commits Enable Safe Rollback

Use atomic commits — one logical change per commit. Claude Code can decompose large tasks into commit-checkpoint sequences automatically when instructed. If an implementation goes wrong mid-way, atomic commits let you git revert precisely without losing unrelated work.

Git Context Claude Code Reads Automatically

ContextHow Claude Code uses it
Current branch name Infers the feature scope; avoids touching unrelated areas
Staged / unstaged diff Understands what is already in progress before adding more
Recent commit messages Matches commit message style and convention
Remote tracking status Knows if the branch is ahead/behind origin before suggesting a push
Parallelism

Parallel Claude Code Sessions

Git worktrees allow multiple simultaneous checkouts of the same repository in different directories. Each worktree is a fully isolated workspace sharing the same git history — enabling you to run multiple Claude Code instances in parallel on different features with zero interference.

Why Worktrees?

Without worktrees, running two Claude Code sessions in the same repository directory causes conflicts: they read and write the same working tree simultaneously. With worktrees, each session has its own directory on disk — separate files, separate unstaged changes — while sharing the underlying git object store.

3x
parallel sessions on the same repo
0
conflicts between sessions
~2hr
typical time saved on parallel feature work

Setting Up Parallel Worktrees

1
Create a worktree for each feature
bash
# From your main repo directory
git worktree add ../feature-auth feature/auth-refactor
git worktree add ../feature-api feature/api-v2
git worktree add ../feature-perf feature/db-optimization

Each command creates a new directory and a new branch checked out into it. The directories live alongside your main repo directory.

2
Open Claude Code in each worktree directory

Open a terminal tab or window for each worktree directory and launch Claude Code in each one:

bash — terminal tab 1
cd ../feature-auth && claude
bash — terminal tab 2
cd ../feature-api && claude
3
Work independently in each session

Each Claude Code instance sees only its worktree's files. Changes in one worktree do not affect the others. Both sessions share the same git history — commits made in one worktree are immediately visible to the other via git log, but the working trees remain separate.

4
Merge back when each feature is complete
bash
# Open PRs from each worktree branch as normal
gh pr create --base main --head feature/auth-refactor

# After merge, remove the worktree
git worktree remove ../feature-auth
git branch -d feature/auth-refactor

Sequential vs Parallel Comparison

Sequential Work (single session)
  • One feature at a time
  • Blocking — waiting on review blocks next feature
  • Linear progression
  • Simple setup, one directory
  • No risk of cross-contamination
Parallel Worktrees (multiple sessions)
  • Multiple features simultaneously
  • Non-blocking — one in review, others in flight
  • Concurrent, faster overall delivery
  • Requires multiple terminal tabs/windows
  • Isolated by design — no interference
Ideal Parallelism Strategy

A practical pattern: have one Claude Code session implementing the current feature, a second session writing tests for the previous feature while it is in review, and a third session doing exploratory research or documentation. All three advance simultaneously without stepping on each other.

Process

The Spec-Driven Loop

The spec-driven loop combines Plan Mode, review, implementation, and testing into a disciplined cycle that prevents the most common failure mode: implementing the wrong thing correctly.

The Complete Loop

1
Enter Plan Mode
Use Shift+Tab+Tab or /plan. Describe the full requirement, constraints, and any non-obvious context. The more precise the input, the more useful the spec output.
2
Review the Spec Output
Read the plan carefully. Verify file list, design decisions, and identified risks. Push back on anything you disagree with — ask Claude Code to reconsider specific decisions before proceeding.
3
Approve and Implement
Exit Plan Mode and tell Claude Code to proceed. It will implement against the approved spec, committing at logical checkpoints.
4
Test
Run the test suite. If failures occur, feed them back to Claude Code with the error output — it will diagnose and fix without requiring you to re-explain the whole task.
5
PR and Review
Claude Code generates the PR description from the git diff and commit history. Human review validates the implementation against the original spec.
Fix the Spec, Not the Implementation

The spec is the source of truth. If you disagree with part of the spec after implementation has begun, stop — roll back, correct the spec, then re-implement. Changing course mid-implementation accumulates technical debt and produces inconsistent code. A two-minute spec correction prevents hours of cleanup.

Spec Quality Checklist

Before approving a plan, verify it contains:

ElementWhy it matters
Explicit list of files to create/modifyBounds scope; prevents Claude Code from touching unrelated code
Key design decisions with rationaleLets you catch wrong choices before code is written
Risks and mitigationsSurfaces edge cases before they become bugs
Explicit out-of-scope itemsPrevents scope creep during implementation
Automation

Headless Mode and the -p Flag

Claude Code can run entirely non-interactively using the -p (print) flag. Headless mode reads a prompt from stdin or as an argument, executes the task, prints the output to stdout, and exits — making it composable with any shell pipeline, CI system, or automation script.

Basic Usage

Pass a prompt directly with -p. Claude Code runs the task, writes all output to stdout, and exits with a zero or non-zero code depending on success:

bashBasic headless invocation
claude -p "Summarize what the function parseConfig does in pkg/config/config.go"
bashPiping a prompt from stdin
echo "List all exported functions in internal/auth/ with a one-line description of each" \
  | claude -p
bashCapturing output in a variable
REVIEW=$(claude -p "Review internal/handlers/user.go for error handling issues. Output as JSON.")
echo "$REVIEW" | jq '.issues[] | select(.severity == "HIGH")'

Piping Claude into Shell Scripts

Because -p writes to stdout, you can pipe Claude Code output into any standard Unix tool:

bashPipeline: Claude output piped to jq
claude -p "Audit src/handlers/ for SQL injection risks. Return JSON: [{file, line, snippet}]" \
  | jq '.[] | "\(.file):\(.line) — \(.snippet)"' \
  | sort
bashUsing output to gate a deploy
ISSUES=$(claude -p "Check for hardcoded secrets or API keys in the staged diff. Output: CLEAN or ISSUES_FOUND")
if [[ "$ISSUES" == "ISSUES_FOUND" ]]; then
  echo "Aborting: Claude found potential secrets in diff"
  exit 1
fi
git push origin main

CI/CD Integration

The -p flag is designed for CI pipelines. Combine it with GitHub Actions to add AI-assisted checks to your pull request workflow:

yaml.github/workflows/claude-review.yml
name: Claude Code Review

on:
  pull_request:
    types: [opened, synchronize]

jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Install Claude Code
        run: npm install -g @anthropic-ai/claude-code

      - name: Review changed files
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          CHANGED=$(git diff --name-only origin/main...HEAD | grep '\.go$')
          for FILE in $CHANGED; do
            claude -p "Review $FILE for: missing error handling, nil dereference risks, and input validation gaps. Output JSON: [{file, line, severity, description}]" >> review.json
          done

      - name: Post review as PR comment
        uses: actions/github-script@v7
        with:
          script: |
            const fs = require('fs')
            const review = fs.readFileSync('review.json', 'utf8')
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: '## Claude Code Review\n\n```json\n' + review + '\n```'
            })
Tip: Always set ANTHROPIC_API_KEY as a GitHub Actions secret, never hardcode it. Use --no-interactive alongside -p in CI to prevent Claude Code from pausing for confirmation prompts.
Note: Headless mode does not display the interactive UI — no spinners, no prompts, no ANSI coloring. Output is plain text suitable for logging. Use --output-format json when you need machine-readable structured output from automated runs.
Automation

Hooks

Hooks are shell commands or scripts that Claude Code runs automatically at specific lifecycle events. They let you automate repetitive actions, enforce team conventions, run tests on every save, play sounds, block dangerous commands, and create a fully customized development loop — all without manually triggering anything.

Hook Types

Claude Code supports five hook types, each firing at a different point in the execution lifecycle:

Hook TypeWhen it firesCommon use
PreToolUse Before Claude calls any tool (Bash, Edit, Write, etc.) Validate or block dangerous tool calls, log every tool invocation
PostToolUse After a tool call completes, before Claude continues Auto-format code after writes, run linting after edits, run tests after file changes
Notification When Claude sends a notification event Send Slack/email alerts, play sounds, show OS desktop notifications
Stop When Claude Code exits or finishes a session Play a completion sound, send a summary notification, log session stats
SubagentStop When a subagent completes its task Log subagent results, trigger post-processing, chain next step

Hook Configuration

Hooks are configured in .claude/settings.json (project-scoped) or ~/.claude/settings.json (user-scoped). Each hook is a shell command string or a path to an executable script:

json.claude/settings.json — hook configuration
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          { "type": "command", "command": "gofmt -w \"$CLAUDE_TOOL_FILE\"" }
        ]
      }
    ],
    "Stop": [
      {
        "hooks": [
          { "type": "command", "command": "uv run play_sound.py" }
        ]
      }
    ]
  }
}

Hook Payloads

When a hook fires, Claude Code injects context about the event into environment variables. Your hook script can read these to make decisions:

Environment VariableAvailable inValue
CLAUDE_TOOL_NAMEPreToolUse, PostToolUseName of the tool called (e.g., Bash, Edit)
CLAUDE_TOOL_INPUTPreToolUse, PostToolUseJSON-encoded input parameters passed to the tool
CLAUDE_TOOL_OUTPUTPostToolUseJSON-encoded output returned by the tool
CLAUDE_TOOL_FILEPostToolUse (Edit/Write)Absolute path of the file that was written or edited
CLAUDE_SESSION_IDAll hooksUnique identifier for the current Claude Code session
CLAUDE_NOTIFICATION_TYPENotificationType of notification event
CLAUDE_NOTIFICATION_MESSAGENotificationNotification message text

PreToolUse: Blocking Dangerous Commands

A PreToolUse hook that exits with a non-zero code will block the tool call entirely. Claude Code receives the exit code and skips the tool. Use this to enforce safety rules:

bash.claude/hooks/guard-bash.sh — blocks destructive commands
#!/usr/bin/env bash
set -euo pipefail

COMMAND=$(echo "$CLAUDE_TOOL_INPUT" | jq -r '.command // ""')

BLOCKED_PATTERNS=(
  "rm -rf /"
  "DROP TABLE"
  "DROP DATABASE"
  "git push --force origin main"
  "git push --force origin master"
  "> /etc"
)

for PATTERN in "${BLOCKED_PATTERNS[@]}"; do
  if echo "$COMMAND" | grep -qi "$PATTERN"; then
    echo "BLOCKED: Refused to run: $COMMAND" >&2
    echo "Pattern matched: $PATTERN" >&2
    exit 1
  fi
done

exit 0
jsonRegister the guard hook
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          { "type": "command", "command": ".claude/hooks/guard-bash.sh" }
        ]
      }
    ]
  }
}

PostToolUse: Auto-Format on Save

Run gofmt or your formatter automatically every time Claude writes or edits a file. No more manually running formatters:

bash.claude/hooks/auto-format.sh
#!/usr/bin/env bash
set -euo pipefail

FILE="$CLAUDE_TOOL_FILE"

case "$FILE" in
  *.go)
    gofmt -w "$FILE"
    goimports -w "$FILE" 2>/dev/null || true
    ;;
  *.ts|*.tsx|*.js)
    npx prettier --write "$FILE" 2>/dev/null || true
    ;;
  *.py)
    black "$FILE" 2>/dev/null || true
    ;;
esac

exit 0

PostToolUse: Run Tests After Changes

Trigger the test suite every time Claude edits a source file to catch regressions immediately:

bash.claude/hooks/run-tests.sh
#!/usr/bin/env bash
set -euo pipefail

FILE="$CLAUDE_TOOL_FILE"

if [[ "$FILE" == *.go ]] && [[ "$FILE" != *_test.go ]]; then
  PACKAGE=$(dirname "$FILE")
  echo "Running tests for $PACKAGE..."
  go test "$PACKAGE/..." -timeout 30s -count=1 2>&1
fi

exit 0

Notification Hook: OS Desktop Alerts

Send a desktop notification when Claude completes a long task, so you can leave Claude running and return when it is done:

bash.claude/hooks/notify.sh — cross-platform desktop notification
#!/usr/bin/env bash

MESSAGE="${CLAUDE_NOTIFICATION_MESSAGE:-Claude Code task complete}"

if command -v osascript &>/dev/null; then
  osascript -e "display notification \"$MESSAGE\" with title \"Claude Code\""
elif command -v notify-send &>/dev/null; then
  notify-send "Claude Code" "$MESSAGE"
fi

exit 0

Logging Hook: Full Audit Trail

Log every tool call with its input and output to a file for audit and debugging purposes:

bash.claude/hooks/audit-log.sh
#!/usr/bin/env bash

LOG_FILE="${HOME}/.claude/audit.log"
TIMESTAMP=$(date -u +"%Y-%m-%dT%H:%M:%SZ")

echo "{\"ts\":\"$TIMESTAMP\",\"session\":\"$CLAUDE_SESSION_ID\",\"tool\":\"$CLAUDE_TOOL_NAME\",\"input\":$CLAUDE_TOOL_INPUT}" \
  >> "$LOG_FILE"

exit 0

✓ Good Hook Design

bash
#!/usr/bin/env bash
set -euo pipefail

FILE="$CLAUDE_TOOL_FILE"

[[ "$FILE" == *.go ]] || exit 0
gofmt -w "$FILE"
exit 0
  • Fast exit when hook does not apply
  • Uses set -euo pipefail for safety
  • Explicit exit 0 on success
  • Single focused responsibility

✗ Anti-Pattern

bash
#!/usr/bin/env bash

gofmt -w .
go test ./...
go vet ./...
golangci-lint run
npm run build
docker build .
  • Runs on every tool call regardless of relevance
  • Slow — blocks Claude Code for minutes
  • No early exit for non-matching files
  • Multiple unrelated concerns in one hook
Warning: Hook scripts that block for a long time (e.g., running the full test suite on every keystroke) will slow Claude Code to a crawl. Keep hooks fast — under 2 seconds for PostToolUse hooks. For slow operations like full builds, use the Stop hook instead, which fires only once at session end.
Automation

Pipeline Patterns

Claude Code's headless mode and composable output enable powerful pipeline patterns — chaining multiple Claude operations, feeding output from one step into the next, and integrating with external tools to build automated workflows far beyond what the interactive session supports.

Chain Pattern: Plan Then Implement

Run Plan Mode headlessly to generate a spec, capture it to a file, then feed it back to Claude to drive implementation. This turns the spec-driven loop into a fully automated pipeline:

bashTwo-stage plan-then-implement pipeline
#!/usr/bin/env bash
set -euo pipefail

TASK="$1"

echo "--- Phase 1: Generating spec ---"
SPEC=$(claude -p "/plan $TASK" --output-format text)
echo "$SPEC"

echo ""
echo "--- Phase 2: Implementing from spec ---"
claude -p "Implement the following plan exactly as specified:

$SPEC

Commit each logical unit as you complete it using conventional commits."

Fan-Out Pattern: Parallel File Processing

Process multiple files in parallel by spawning one Claude process per file, then aggregating results:

bashParallel Claude processes — one per Go file
#!/usr/bin/env bash
set -euo pipefail

RESULTS_DIR=$(mktemp -d)
PIDS=()

for FILE in internal/handlers/*.go; do
  (
    claude -p "Review $FILE for missing error handling. Output JSON: [{file,line,issue}]" \
      > "$RESULTS_DIR/$(basename $FILE).json"
  ) &
  PIDS+=($!)
done

for PID in "${PIDS[@]}"; do
  wait "$PID"
done

jq -s 'flatten' "$RESULTS_DIR"/*.json | jq 'sort_by(.file, .line)'
rm -rf "$RESULTS_DIR"

Validation Gate Pattern

Use Claude as a quality gate in your commit or deploy process. Block progression if Claude finds issues:

bash.git/hooks/pre-push — Claude as push gate
#!/usr/bin/env bash
set -euo pipefail

CHANGED_FILES=$(git diff --name-only origin/main...HEAD | grep '\.go$' || true)

if [[ -z "$CHANGED_FILES" ]]; then
  exit 0
fi

RESULT=$(claude -p "Check these files for critical issues (nil dereferences, missing error handling, hardcoded credentials): $CHANGED_FILES. Output: PASS or FAIL:")

if [[ "$RESULT" == FAIL* ]]; then
  echo "Claude Code push gate failed: $RESULT"
  exit 1
fi

echo "Claude Code push gate: PASS"
exit 0
Note: Pipeline scripts depend on predictable Claude output format. Always specify the exact output format in your headless prompt (e.g., "Output JSON array" or "Output: PASS or FAIL:reason"). Vague format instructions produce vague output that is difficult to parse reliably in scripts.
