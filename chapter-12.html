Chapter 12 Overview

What Are Deep Agents?

Deep agents are agents that maintain long-running state, reason about their own execution, and self-direct complex multi-step work. They are fundamentally different from shallow agents — one-shot responders that answer questions and stop. Claude Code is a deep agent.

The Core Distinction

When you ask a shallow agent "how do I implement OAuth?", it answers and stops. When you ask Claude Code "implement OAuth in this repository", it decomposes the task, writes a todo list, reads the codebase, implements code across multiple files, runs tests, fixes failures, and marks each step complete as it goes. That is a deep agent: persistent, self-directing, and capable of recovering from failure.

Claude Code exhibits all the hallmarks of a deep agent:

  • It maintains a todo list that persists across steps
  • It tracks its own progress and backtracks when stuck
  • It spawns subagents for subtasks that can run in parallel
  • It uses the file system as persistent memory between steps

What Makes a Task "Deep"

Not every task needs a deep agent approach. A task is "deep" when it exhibits one or more of these characteristics:

Characteristic Shallow Task Deep Task
Scope One file or one function Multiple files, packages, or services
Duration Seconds to a few minutes Minutes to hours
Verification Visual inspection Running tests, building, integration checks
Unknowns Requirements are fully clear upfront Discovery is needed — read code to understand before writing
Failure handling User corrects and retries Agent self-corrects and continues
Tip: Give deep agents goals, not step-by-step instructions. "Implement JWT refresh token rotation with all tests passing" is a deep agent goal. "In jwt.go, add a field called RefreshToken to the User struct" is an instruction better suited for a shallow, direct request. Deep agents excel at figuring out the steps — that is their core value.

Concepts

Agent Taxonomy: Deep vs Shallow

The distinction between shallow and deep agents is not about intelligence — it is about architecture. Shallow agents are stateless request-response systems. Deep agents are stateful autonomous systems that persist across many reasoning steps.

Characteristic Shallow Agent Deep Agent
Task duration Single response Multi-step, minutes to hours
State Stateless Maintains todo list, progress
Error handling Returns error to user Backtracks, retries, self-corrects
Tool use Limited or none Extensive (file, shell, search)
Delegation None Spawns subagents
Example ChatGPT Q&A Claude Code implementing a feature
A Fundamental Architecture Difference

Most "AI assistants" are shallow agents. Claude Code's architecture is fundamentally different — it is designed to execute tasks autonomously over long horizons. Understanding this distinction helps you work with it correctly: give it goals, not instructions, and let it reason about the steps.

Internals

How Claude Code Uses TODOs Internally

Claude Code maintains an internal todo list that you can inspect with /todo. This is not a cosmetic feature — it is the mechanism by which the agent decomposes work, tracks progress, and backtracks when it hits blockers.

The Task Execution Loop

1
Decompose the task
Claude Code breaks the high-level goal into concrete, ordered steps based on its understanding of the codebase and the task requirements.
2
Write steps to the todo list
Each step is written as a discrete todo item with a clear, actionable description. Nested subtasks are indented under their parent step.
3
Execute steps sequentially (or in parallel)
Steps that depend on each other run sequentially. Independent steps can be delegated to subagents and run in parallel to save time.
4
Mark items complete as it goes
Each completed step is marked immediately. This creates a clear record of what has been done and what remains, visible at any point via /todo.
5
Backtrack and create new todos when blocked
If a step fails or reveals new complexity, Claude Code revises the plan — marking the blocker and creating new todo items to address it before continuing.

What a Claude Code Todo List Looks Like

text
[✓] Read existing auth middleware (src/auth/middleware.go)
[✓] Understand current JWT validation approach
[→] Implement refresh token rotation
   [→] Add RefreshToken field to User struct
   [ ] Write database migration
   [ ] Update JWT service
   [ ] Add refresh endpoint
[ ] Write tests for refresh token flow
[ ] Update documentation
Tip: You can inspect and modify Claude Code's todo list at any point during a task. If it goes off track — adding unnecessary steps or missing important ones — you can reset or edit the todos to refocus it. This is one of the most powerful levers for steering a deep agent mid-execution.

The GSD (Get Stuff Done) Pattern

For complex multi-step tasks, structure your initial prompt using the Plan → Research → Execute → Verify cycle. This mirrors how an expert engineer approaches a large task:

textGSD prompt structure for deep tasks
Goal: Implement rate limiting on all public API endpoints

Phase 1 — Research (do this first, do not write code yet):
- Read the existing middleware chain in pkg/middleware/
- Identify all router files that register public endpoints
- Check if any rate limiting library is already in go.mod
- Document findings before proceeding

Phase 2 — Plan (write the implementation plan as a todo list):
- Based on research, create a concrete todo list
- Each item must be a discrete, testable change
- Get confirmation before proceeding to Phase 3

Phase 3 — Execute:
- Implement each todo item
- Run `go test ./...` after each file change
- Do not proceed to the next item if tests fail

Phase 4 — Verify:
- Run the full test suite
- Run `go build ./...`
- Write a summary of what was changed and what tests cover it

The explicit phase structure prevents a common failure mode: the agent charges into writing code before fully understanding the codebase, producing changes that conflict with existing patterns or miss important integration points.

Architecture

Hierarchical Delegation

Deep agents delegate to subagents, which can further delegate — creating a tree of agents where each level handles increasing specificity. The orchestrator manages the goal, coordinators manage features, and workers handle individual files or atomic tasks.

The Delegation Tree

For a large feature implementation, Claude Code might create a delegation structure that looks like this:

How Delegation Works

  • Orchestrator — receives the high-level goal, decomposes it into major workstreams, delegates each to a coordinator subagent
  • Coordinators — receive a scoped workstream (e.g., "implement the auth feature"), decompose it into atomic file-level tasks, delegate to workers
  • Workers — receive a single, well-defined task (e.g., "add RefreshToken field to User struct in auth.go"), execute it, return structured results
Warning: Deep hierarchies — three or more levels — can lose context at each delegation step. Each boundary compresses information; summaries replace raw content. Prefer shallow delegation (two levels max) with rich, detailed task descriptions at each handoff. The quality of what workers produce is only as good as the context they receive.

Multi-Phase Project Pattern

For large projects that span multiple sessions, break the work into phases with explicit verification gates between them. Each phase must produce a verifiable artifact before the next phase begins:

markdownCLAUDE.md — multi-phase project structure
## Current Project: API v2 Migration

### Phase 1 — Schema Design (COMPLETE)
- New schema documented in docs/api-v2-schema.md
- Breaking changes listed in docs/api-v2-breaking-changes.md
- Gate: schema approved in PR #142

### Phase 2 — Implementation (IN PROGRESS)
- Implementing endpoints per docs/api-v2-schema.md
- Each endpoint needs: handler, validation, tests
- Gate: all existing tests pass + new tests added

### Phase 3 — Migration (BLOCKED on Phase 2)
- Write data migration scripts
- Dual-write period for backward compatibility
- Gate: load test confirms no regression

### Do not start a phase before the previous gate is cleared.

This structure in CLAUDE.md means every session — including new subagent sessions — starts with the current project state. The agent knows what phase it is in, what is done, and what the acceptance criteria for the current phase are.

Concepts

Context Flow in Deep Agent Systems

Context flows in two directions: downward from orchestrator to worker (task + relevant code), and upward from worker to orchestrator (results + summaries). Each delegation boundary compresses information — understanding this compression is key to effective deep agent design.

What to Pass at Each Boundary

Context Direction What to Include What to Omit
Downward (to worker) Task description, relevant file paths, code snippets, acceptance criteria, constraints Unrelated conversation history, verbose context from previous steps, implementation details not relevant to the subtask
Upward (from worker) Structured findings (JSON/markdown), specific changes made, file paths and line references, blockers encountered Implementation details not requested, intermediate reasoning steps, raw tool call outputs

Why Compression Matters

Each context window is finite. At each delegation boundary, the orchestrator must decide what is essential to pass down. A coordinator that receives 50,000 tokens of context but only needs 5,000 tokens to do its job is wasting capacity — capacity that could be used for the actual work. Good deep agent design front-loads compression: include only what the subagent needs to succeed, nothing more.

Structured Outputs Are Compressed Context

Ask subagents to return structured results — JSON objects or markdown with specific headings — rather than free-form prose. Structured outputs are easier to parse, easier to route, and more reliably summarize exactly what happened. A worker returning {"changed_files": ["auth.go"], "lines_added": 42, "blockers": []} is far more useful to an orchestrator than three paragraphs of explanation.

Patterns

File System as Agent Memory

Deep agents use the file system as persistent memory between steps. Reading files provides current state; writing files communicates progress and intermediate results. This is why CLAUDE.md is so powerful — it is memory that persists across sessions and survives context resets.

The Memory Pattern

1
Agent reads CLAUDE.md for project context
At the start of every session, Claude Code reads CLAUDE.md. Architecture decisions, known issues, coding guidelines, and project conventions all live here — and the agent follows them without being told explicitly each time.
2
Agent reads existing code to understand current state
Before making changes, the agent reads the files it will modify. This grounds it in the actual current state of the codebase, not a stale mental model from earlier in the session.
3
Agent writes changes incrementally
Changes are written file by file, not all at once. Each write is a discrete, reviewable step. This incremental approach also means partial work survives if the session is interrupted.
4
Agent writes notes or summaries for later steps
For long multi-step tasks, the agent may write intermediate summaries to temp files. Later steps — or subagents — read these files to pick up where the previous step left off.
5
Subagents read these files for context
When an orchestrator delegates to a worker, the worker reads the relevant files directly rather than receiving a copy of the content in its prompt. This keeps delegation context small and always fresh.

CLAUDE.md as Persistent Memory

Unlike conversation context — which resets between sessions — CLAUDE.md persists. Every session starts by reading it. This makes it the right place to store anything you want Claude Code to know permanently:

  • Architecture decisions — why certain patterns were chosen, what alternatives were rejected
  • Current state — what is in progress, what is blocked, what was recently changed
  • Known issues — bugs you are aware of, workarounds in place, things to avoid
  • Guidelines — code style, testing requirements, PR conventions
Tip: Use clear section headings: Architecture, Current State, Known Issues, and Guidelines. Agents read CLAUDE.md every session and follow the guidance it contains. The more specific and actionable the content, the more reliably the agent will behave as you expect — across sessions, after context resets, and when spawning subagents that inherit the same file.
Resilience

Recovery from Failure

Long-running tasks will encounter failures — test breakage, unexpected code patterns, ambiguous requirements. Deep agents handle most failures autonomously, but understanding the recovery mechanisms helps you intervene effectively when needed.

How Claude Code Self-Recovers

When Claude Code hits a failure during a task, it follows a standard recovery pattern:

1
Identify the failure
Read the error output from the failed command. Identify whether it is a compilation error, test failure, permission error, or unexpected output.
2
Diagnose root cause
Read the relevant source files to understand why the failure occurred. Does the existing code conflict with the new change? Is a dependency missing?
3
Revise the plan
Add new todo items to address the root cause. Mark the current item as blocked. Create a corrected approach before retrying.
4
Apply fix and verify
Make the corrective change and re-run the failing check. Verify the fix resolves the failure without introducing new ones.
5
Continue the original task
Resume the original plan from where it left off, now with the underlying issue resolved.

When to Intervene Manually

Some failures require human judgment. Intervene when you observe:

  • The agent has retried the same approach more than twice without success
  • The failure suggests a fundamental misunderstanding of the codebase architecture
  • The fix requires a design decision — not just a technical correction
  • The agent is changing code outside the original task scope to work around a failure
Tip: When you need to redirect a failing agent, do not just say "that approach is wrong." Explain why it is wrong and provide the correct constraint. "Don't modify the User struct directly — it is generated code. The source of truth is in schema.graphql" gives the agent the information it needs to find the right path.

Rollback Strategies

If an agent makes incorrect changes that you want to undo, use git:

bashRolling back agent changes
# See what the agent changed
git diff HEAD

# Undo uncommitted changes to specific files
git checkout -- src/auth/jwt.go

# Undo all uncommitted changes (use with caution)
git checkout -- .

# If agent made commits, revert specific commits
git revert HEAD

# Or reset to a known-good state
git reset --hard origin/main
Note: This is why you should always work on a branch when running deep agents on significant tasks. A branch gives you a clean rollback path: if the agent produces incorrect results, you can simply delete the branch and start fresh. Never run long deep-agent tasks directly on main.
Operations

Cost Management for Deep Tasks

Deep agents consume significantly more tokens than shallow requests. A complex multi-file task can use 10-50x the tokens of a simple question. Understanding where tokens go — and how to control costs — is essential for using deep agents at scale.

Where Tokens Go

Source Token Volume Reduction Strategy
File reads (context building) High — every file read adds to context Use .claudeignore to exclude irrelevant files
Conversation history Grows with every step Use /clear between independent tasks; start fresh sessions
Tool call results Medium — bash output, file contents Write scripts that return only relevant output (not raw logs)
Backtracking and retries Variable — depends on failure rate Clear task descriptions reduce ambiguity and retries

Practical Cost Controls

  • Create a .claudeignore file — exclude build artifacts, vendor directories, generated files, and test fixtures that Claude Code does not need to read
  • Break large tasks into sessions — a 4-hour task is more expensive than four 1-hour tasks, because context accumulates. Clear between phases.
  • Front-load information in CLAUDE.md — every token you save Claude from re-discovering via file reads is a token saved
  • Scope tasks precisely — "implement X in pkg/auth/" is cheaper than "implement X across the codebase" because Claude reads fewer files
